1. Unit Tests — Built In, Zero Setup

JUnit has had 5 major versions, each with different annotations, runners, and config files. JUnit 4 uses @Test from one package. JUnit 5 uses @Test from a different package — same name, different import, different behaviour. The test for a correct JUnit 5 setup is whether your tests actually run. Rust ships #[test] built into the language. No dependencies, no config, no JAR files. Just write the test.
// src/math.rs
pub fn add(a: i32, b: i32) -> i32 { a + b }
pub fn divide(a: f64, b: f64) -> Option<f64> {
    if b == 0.0 { None } else { Some(a / b) }
}

#[cfg(test)]          // compile ONLY during test runs
mod tests {
    use super::*;     // import everything from parent module

    #[test]
    fn test_add() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn test_add_negative() {
        assert_eq!(add(-1, -1), -2);
    }

    #[test]
    fn test_divide_by_zero() {
        assert_eq!(divide(10.0, 0.0), None);
    }

    #[test]
    fn test_divide() {
        assert!((divide(10.0, 3.0).unwrap() - 3.333).abs() < 0.001);
    }

    #[test]
    #[should_panic(expected = "called unwrap on None")]
    fn test_unwrap_none_panics() {
        divide(1.0, 0.0).unwrap(); // panics — test passes
    }
}

Run tests:

cargo test                          # run all tests
cargo test test_add                 # run tests matching "test_add"
cargo test -- --nocapture           # show println! output during tests
cargo test -- --test-threads=1      # run serially (no parallelism)

should_panic — the test passes if the function panics, fails if it returns normally. You can be more specific: #[should_panic(expected = "divide by zero")] — the panic message must contain that string. This tests your error paths explicitly, unlike C# where you'd use Assert.Throws<Exception>().
A well-tested codebase with #[test] coverage, property tests, and integration tests is genuinely more secure — you catch logic errors before attackers do. // tests/soap_integration.rs

✅ True or False
Code examples in Rust doc comments (between ``` fences) are compiled and run by cargo test.
True! Doc tests are a first-class feature. cargo test runs them automatically. This means your documentation examples can never get out of sync with the actual API — if you rename a function and forget to update the docs, the doc test fails. It's documentation-driven development baked into the toolchain.
In Agile, 100% test coverage is a metric that proves nothing except that your tests run. The coverage report shows green. Production goes down because the test mocked the database and the real database has a column type mismatch from a migration that was 'just a small change'. The PM shows the coverage graph to the client during the sprint review. Everyone claps.

4. Cargo Features — Conditional Compilation at Scale

// Cargo.toml — define optional features:
[features]
default = ["http"]           # features enabled by default
http = ["reqwest"]           # "http" feature pulls in reqwest dep
metrics = ["prometheus"]     # optional metrics
dev-tools = []               # no extra deps, just a flag

[dependencies]
reqwest = { version = "0.11", optional = true }
prometheus = { version = "0.13", optional = true }

// In code — use #[cfg(feature = "...")] to gate code:
#[cfg(feature = "metrics")]
mod metrics {
    pub fn record_request_duration(ms: u64) {
        HISTOGRAM.observe(ms as f64 / 1000.0);
    }
}

#[cfg(not(feature = "metrics"))]
mod metrics {
    pub fn record_request_duration(_ms: u64) {}  // no-op
}

// Enable features when building:
// cargo build --features metrics
// cargo build --features "metrics,dev-tools"
// cargo build --all-features
// cargo build --no-default-features
SAP's testing story: there isn't one. ABAP Unit exists since NW 7.0, but the vast majority of SAP customising is tested by clicking through the transaction in DEV and saying 'looks fine'. The transport moves to QA. QA does the same clicking. The transport moves to PROD on a Friday. The MD07 transaction crashes Monday morning and everyone acts surprised.
🔘 Multiple Choice
What's the purpose of #[cfg(feature = "dev-tools")] on a function?
✓ Conditional compilation! The function literally doesn't exist in binaries built without that feature. This is how you ship debug panels, test endpoints, or expensive instrumentation that should never reach production. No runtime check — the code is physically absent. Equivalent to C# preprocessor directives (#if DEBUG) but more granular and managed through Cargo's dependency system.

5. Cargo Workspaces — Monorepos

# workspace Cargo.toml (crate root)
[workspace]
members = [
    "apinox-core",
    "apinox-tauri",
    "apinox-cli",
    "shared-types",
]

# Each member has its own Cargo.toml
# All members share a single Cargo.lock
# Single target/ directory (build artifacts deduped)
# apinox-tauri/Cargo.toml — refer to sibling crates:
[dependencies]
apinox-core = { path = "../apinox-core" }
shared-types = { path = "../shared-types" }
# Cargo commands work across the whole workspace:
cargo build                    # build everything
cargo test                     # test everything
cargo test -p apinox-core      # test only one member
cargo build -p apinox-cli      # build only one member
APInox doesn't use a workspace — it's a single crate. But if you were to split the proxy/mock module logic into a pure Rust library and keep the Tauri-specific code separate, a workspace would be the right tool. It's the Rust equivalent of a .NET solution with multiple projects.
🔗 Match It
Match the Cargo command to its purpose:
All matched!
• cargo check → type-checks your code without full compilation — 5-10× faster than cargo build, great for tight edit-check loops
• cargo clippy → Rust's official linter — catches common mistakes and suggests idiomatic improvements
• cargo doc --open → generates beautiful HTML docs from your code and opens in browser
• cargo add tokio → adds to Cargo.toml automatically (Cargo 1.62+), like npm install
🐛 Spot the Bug
This test always passes, but it has a logic error. Click the buggy line:
#[test] fn test_parse() {     let result = parse("42");     assert!(result.is_ok()); }
Line 4! assert!(result.is_ok()) checks that parsing succeeded — but it doesn't check the value. parse("42") could return Ok(-1) and the test would still pass. Better: assert_eq!(result.unwrap(), 42); — this verifies both success AND the correct parsed value. Testing that something doesn't error is a starting point, but test the actual output too.
