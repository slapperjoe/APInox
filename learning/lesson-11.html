1. Display and Debug — Two Ways to Print

TraitFormatForC# analogy
Debug{:?} / {:#?}Developers — verbose, automaticDebugger watch window

// Debug — just derive it:
#[derive(Debug)]
struct SoapRequest {
    endpoint: String,
    action: String,
}
println!("{:?}", req);   // SoapRequest { endpoint: "...", action: "..." }
println!("{:#?}", req);  // pretty-printed with newlines

// Display — implement it yourself:
use std::fmt;

impl fmt::Display for SoapRequest {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "POST {} [{}]", self.endpoint, self.action)
    }
}
println!("{}", req);     // POST https://... [MyAction]
// Also enables: req.to_string(), format!("{}", req)
✅ True or False
Implementing Display for a type automatically gives it a .to_string() method.
True! The standard library has a blanket implementation: for any type that implements Display, ToString is also implemented. So impl Display gives you .to_string() for free. This is why you call error.to_string() on errors — errors implement Display.

2. From and Into — Conversion Traits

These are the idiomatic way to convert between types. Implement one, get the other free:

// Implement From:
impl From<String> for SoapEndpoint {
    fn from(url: String) -> Self {
        SoapEndpoint { url, timeout_ms: 30_000 }
    }
}

// You get Into for free via a blanket impl:
// impl<T, U: From<T>> Into<U> for T

// Both of these now work:
let ep = SoapEndpoint::from("https://...".to_string());
let ep: SoapEndpoint = "https://...".to_string().into();

// Very common in function signatures — accept anything convertible:
fn connect(endpoint: impl Into<SoapEndpoint>) {
    let ep = endpoint.into();  // convert at call site
}
connect("https://api.example.com".to_string());  // works
connect(my_endpoint);                             // works — no-op From

TryFrom / TryInto — Fallible Conversions

use std::convert::TryFrom;

impl TryFrom<&str> for Port {
    type Error = String;
    fn try_from(s: &str) -> Result<Self, String> {
        let n: u16 = s.parse().map_err(|_| format!("Not a valid port: {}", s))?;
        if n == 0 { return Err("Port 0 is reserved".to_string()); }
        Ok(Port(n))
    }
}

let p = Port::try_from("8080")?;
let p: Port = "8080".try_into()?;  // TryInto provided automatically
🔘 Multiple Choice
If you implement From<String> for MyType, what do you get for free?
✓ Into<MyType> for String! The standard library has: impl<T, U> Into<U> for T where U: From<T>. This means: "for any T, if U can be created From T, then T can be converted Into U." So implement From, get Into free. The standard advice: always implement From, never directly implement Into.

3. Default — Zero-Argument Construction

// Derive Default — uses Default for each field:
#[derive(Default)]
struct Config {
    timeout_ms: u64,       // default: 0
    retries: u32,          // default: 0
    verbose: bool,         // default: false
    name: String,          // default: ""
    endpoint: Option<String>, // default: None
}

// Custom Default:
impl Default for Config {
    fn default() -> Self {
        Self {
            timeout_ms: 30_000,  // 30 seconds — not zero!
            retries: 3,
            verbose: false,
            name: String::new(),
            endpoint: None,
        }
    }
}

// Usage:
let config = Config::default();

// Or use struct update syntax — fill in what you need, default the rest:
let config = Config {
    endpoint: Some("https://api.example.com".to_string()),
    ..Config::default()   // fill in timeout, retries, etc. from default
};

It's similar to object spread in JavaScript: { endpoint, ...defaults }. The values are moved from the source, so this is zero-cost — no copying of the whole struct.

4. Blanket Implementations — Traits for All Types

A blanket implementation implements a trait for all types meeting some bound. The standard library does this extensively:

// From std — if you impl Display, you get ToString free:
impl<T: fmt::Display> ToString for T {
    fn to_string(&self) -> String {
        format!("{}", self)
    }
}

// From std — if U implements From<T>, T gets Into<U> free:
impl<T, U: From<T>> Into<U> for T {
    fn into(self) -> U { U::from(self) }
}

// You can write your own:
trait Summary {
    fn summary(&self) -> String;
}
// Blanket — give all Summary types a tweet() method:
trait Tweetable: Summary {   // Supertrait — requires Summary
    fn tweet(&self) -> String {
        format!("📢 {}", self.summary())
    }
}
Supertraits — a trait can require other traits. trait Tweetable: Summary means: to implement Tweetable, you must also implement Summary. Like interface inheritance in C#.

#[derive(Debug, Clone, PartialEq)]
struct Point { x: i32, y: i32 }

let p1 = Point { x: 1, y: 2 };
let p2 = p1.clone();
println!("{}", p1 == p2);
println!("{:?}", p1);
Yes, it compiles! Output:
true
Point { x: 1, y: 2 }

#[derive(Clone)] gives .clone(). #[derive(PartialEq)] gives == (compares field by field). #[derive(Debug)] gives {:?} formatting. All three are generated at compile time — no runtime overhead compared to hand-writing them.
