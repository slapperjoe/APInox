1. The #[tauri::command] Attribute

Any Rust function marked #[tauri::command] becomes callable from the webview via JavaScript's invoke(). Tauri handles:

  • Deserialising the JSON arguments into Rust types
  • Calling your function on a thread-pool thread (for async) or directly (for sync)
  • Serialising the Ok(value) result back to JSON, or forwarding the Err(msg) as a JS rejection
A Tauri command call: TypeScript → IPC → Rust function → JSON result back to TypeScript Promise
// soap/commands.rs — minimal command:
#[tauri::command]
pub async fn cancel_request() -> Result<serde_json::Value, String> {
    log::debug!("Request cancellation called");
    Ok(serde_json::json!({ "success": true }))
}

// ↕ webview TypeScript:
import { invoke } from '@tauri-apps/api/core';
const result = await invoke('cancel_request');
// result = { success: true }

The command name is auto-derived from the function name: cancel_request → "cancel_request". You can override it with #[tauri::command(rename = "cancelRequest")].

Oracle Siebel's command model: every user action goes through a Business Component, hits a Business Object, fires Business Service scripts, writes to 12 tables, raises a workflow event, and commits. A button click to mark a service request as 'In Progress' takes 4 database round trips and 800ms. Enterprise users have learned to click slowly. Siebel calls this 'transactional integrity'. The users call it something else.
✅ True or False
A #[tauri::command] function MUST be async to be callable from the webview.
False! Tauri commands can be sync or async. Sync commands run on the same thread as the event loop (fast, non-blocking operations only). Async commands get tokio::spawn-ed onto a thread pool — use these for anything that does I/O (file reads, HTTP calls, WSDL parsing). In practice, APInox makes almost all commands async to keep the event loop free.

2. Arguments — Automatic JSON Deserialization

Tauri deserializes incoming JSON arguments directly into your function parameter types. If the type implements serde::Deserialize, it works:

// testing/commands.rs — a real APInox command
#[derive(Debug, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct RunTestCaseRequest {
    pub test_case: FrontendTestCase,
    pub variables: Option<HashMap<String, String>>,
    #[serde(default)]
    pub stream: bool,
    pub fallback_endpoint: Option<String>,
}

#[tauri::command]
pub async fn run_test_case(
    request: RunTestCaseRequest,   // ← Tauri deserializes JSON → RunTestCaseRequest
) -> Result<RunTestCaseResponse, String> {
    log::info!("[run_test_case] Starting: '{}' (stream={})",
        request.test_case.name, request.stream);

    if request.stream {
        let run_id = generate_run_id();
        tokio::spawn(async move {
            run_steps_and_stream_updates(&run_id, request).await;
        });
        return Ok(RunTestCaseResponse { run_id: Some(run_id), .. });
    }
    // ... non-streaming path
}
// TypeScript side — camelCase matches thanks to #[serde(rename_all)]
await invoke('run_test_case', {
    request: {
        testCase: { id: '1', name: 'Login test', steps: [...] },
        stream: true,
        variables: { 'baseUrl': 'https://api.example.com' },
        fallbackEndpoint: null
    }
});
✍️ Fill in the Blank
The serde attribute that converts Rust's snake_case fields to JSON camelCase automatically:
#[serde()]
pub struct MyRequest {
    pub test_case: TestCase, // → "testCase" in JSON
    pub run_id: String, // → "runId" in JSON
}
rename_all = "camelCase" — this is essential for Tauri commands because JavaScript conventions use camelCase but Rust conventions use snake_case. Without it you'd have to either use snake_case in TypeScript (ugly) or manually #[serde(rename = "testCase")] every field. The attribute handles the whole struct at once.

3. Managed State — Shared Data Between Commands

Tauri's State<T> is how you share data across commands without global variables. Think of it like ASP.NET Core's dependency injection — you register a type once, and commands receive it automatically.

// 1. Define your state type
struct AppState {
    request_count: Mutex<u64>,
    active_connections: Arc<RwLock<HashMap<String, Connection>>>,
}

// 2. Register it in setup (or .manage() on the builder)
tauri::Builder::default()
    .setup(|app| {
        app.manage(AppState {
            request_count: Mutex::new(0),
            active_connections: Arc::new(RwLock::new(HashMap::new())),
        });
        Ok(())
    })
    // ...

// 3. Access it in any command via State<T> parameter:
#[tauri::command]
pub async fn execute_soap_request(
    request: SoapRequest,
    state: tauri::State<'_, AppState>,  // ← injected automatically
) -> Result<SoapResponse, String> {
    // Increment counter (Mutex guards against race conditions)
    *state.request_count.lock().unwrap() += 1;
    // Execute the request...
    perform_soap_call(request).await
}
APInox uses a simpler pattern: Lazy<Arc<Mutex<HashMap>>> static variables in each module. Both approaches work. State<T> is more idiomatic and testable; statics are quick to write.
SAP's approach to global state: ABAP memory (EXPORT TO MEMORY ID), SAP memory (SET PARAMETER ID), global variables in function groups, and SPA/GPA parameters that persist across transactions for the life of the user session. None of these are thread-safe. ABAP is single-threaded per work process. The application server has 40 work processes. They share nothing. This is SAP's 'stateless architecture'.
🔘 Multiple Choice
Why does managed state typically wrap data in Mutex<T> or RwLock<T>?
Concurrent commands! Every async Tauri command runs on a tokio thread pool — two commands can execute simultaneously. If they both write to the same data without synchronization, you have a data race (undefined behavior). Mutex ensures only one command holds the write lock at a time. RwLock allows multiple simultaneous readers but exclusive writers. The Rust compiler enforces that all shared mutable state goes through one of these mechanisms — you can't accidentally create a data race.

4. AppHandle — The God Object (Used Sparingly)

tauri::AppHandle gives you access to the whole application at runtime — emit events, spawn windows, access managed state, get app directories. You can inject it as a parameter too:

// lib.rs — AppHandle used to manage the app lifecycle:
#[tauri::command]
fn quit_app(app: tauri::AppHandle) {
    log::info!("Quit command received");
    stop_sidecar();
    app.exit(0);   // ← terminate the process cleanly
}

// Or emit an event to ALL webview windows:
#[tauri::command]
pub async fn send_notification(
    app: tauri::AppHandle,
    message: String,
) -> Result<(), String> {
    app.emit("notification", &message)
        .map_err(|e| e.to_string())
}

// Or access a specific window:
#[tauri::command]
pub async fn focus_main_window(app: tauri::AppHandle) {
    if let Some(window) = app.get_webview_window("main") {
        window.set_focus().ok();
        window.show().ok();
    }
}

Use AppHandle when you need to:

  • Emit events to the frontend
  • Access the app's data/config directories
  • Manage window visibility
  • Access managed state from outside a command (in background tasks)
🐛 Spot the Bug
This command should return the number of active connections. Click the buggy line:
static CONNECTIONS: Lazy<Mutex<Vec<String>>> = Lazy::new(|| Mutex::new(vec![])); #[tauri::command] pub fn get_connection_count() -> usize {     CONNECTIONS.lock().len() }
Line 5! .lock() returns a Result<MutexGuard, PoisonError> — you need to .unwrap() (or handle the error) before calling .len(). Fix: CONNECTIONS.lock().unwrap().len(). A Mutex is "poisoned" if a thread panics while holding it, which is why lock() returns a Result rather than the guard directly.

5. The Streaming Pattern — APInox's Test Runner

Long-running operations (test suites, WSDL fetches) can't just block for 30 seconds. APInox solves this with a poll-based streaming pattern:

// testing/commands.rs — the full pattern (simplified):
static TEST_RUN_STORE: Lazy<Arc<Mutex<HashMap<String, TestRunData>>>> = 
    Lazy::new(|| Arc::new(Mutex::new(HashMap::new())));

// Step 1: Start the run, return an ID immediately
#[tauri::command]
pub async fn run_test_case(request: RunTestCaseRequest) -> Result<RunTestCaseResponse, String> {
    let run_id = format!("tc-{}-{}", timestamp_ms(), short_uuid());
    
    // Insert empty slot
    TEST_RUN_STORE.lock().unwrap().insert(run_id.clone(),
        TestRunData { updates: vec![], done: false, error: None });
    
    // Spawn background task — doesn't block the command response
    let run_id_clone = run_id.clone();
    tokio::spawn(async move {
        // Runs steps, pushes updates to TEST_RUN_STORE as it goes...
        for step in &request.test_case.steps {
            let result = run_step(step).await;
            push_update(&run_id_clone, json!({ "type": "stepResult", "result": result }));
        }
        // Mark done
        TEST_RUN_STORE.lock().unwrap()
            .entry(run_id_clone)
            .and_modify(|d| d.done = true);
    });
    
    Ok(RunTestCaseResponse { run_id: Some(run_id), passed: None, duration_ms: None })
}

// Step 2: Frontend polls this to get updates
#[tauri::command]
pub async fn get_test_run_updates(
    run_id: String,
    from_index: usize,
) -> Result<serde_json::Value, String> {
    let store = TEST_RUN_STORE.lock().unwrap();
    let data = store.get(&run_id)
        .ok_or_else(|| format!("Run ID '{}' not found", run_id))?;
    
    Ok(json!({
        "updates": &data.updates[from_index..],
        "done": data.done,
        "error": data.error,
    }))
}
This is a long polling pattern. An alternative is Tauri events (covered in Lesson 19) which push updates to the webview without polling. Events are simpler; the poll pattern is used here because it was migrated from a REST sidecar design.
🔮 Predict & Reveal
What happens if you call run_test_case from TypeScript and the test has 10 steps each taking 2 seconds?
// TypeScript:
const response = await invoke('run_test_case', { request });
console.log('Got response:', response);
// ... how long does this await take?
The await completes almost instantly! The command spawns a tokio::spawn background task and immediately returns { runId: "tc-123-abc" } — before any steps execute. The 10 steps run asynchronously in the background, storing results in TEST_RUN_STORE. The TypeScript side then polls get_test_run_updates every ~500ms to show per-step progress. The total wall time is ~20 seconds, but the command responds in <1ms.
