1. Structs — Data Without Behaviour

In C#, a class holds both data AND methods. In Rust, they're separated: struct holds data, impl adds behaviour (covered in Lesson 3).

Open parsers/wsdl/types.rs:

pub struct SchemaNode {
    pub name: String,
    #[serde(rename = "type")]
    pub node_type: String,
    pub kind: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub min_occurs: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub children: Option<Vec<SchemaNode>>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub is_optional: Option<bool>,
}
// C# equivalent
public class SchemaNode {
    public string Name { get; set; }
    [JsonPropertyName("type")]
    public string NodeType { get; set; }
    [JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)]
    public string? MinOccurs { get; set; }
    public List<SchemaNode>? Children { get; set; }
}
Field visibility: Fields are private by default in Rust. You need pub to expose them — the opposite of C# where fields are private unless you add public... wait, that's the same! The difference is Rust never has a default-public field.

2. Option<T> — No Such Thing as null

Rust has no null. Instead, a value that might be absent is wrapped in Option<T>:

// Option<T> is just this enum under the hood:
enum Option<T> {
    Some(T),   // value is present
    None,      // value is absent
}

// Using it:
let x: Option<String> = Some("hello".to_string());
let y: Option<String> = None;

// Reading it safely:
if let Some(val) = x {
    println!("{}", val);   // only runs if Some
}
🔘 Multiple Choice
What is Option::None equivalent to in C#?
✓ null on a nullable type! Option<String> ≈ string? in C#. The key difference: in Rust, the compiler forces you to handle the None case before you can use the value. In C# you can accidentally dereference a null.

3. Derive Attributes — Free Code Generation

#[derive(...)] tells the compiler to auto-generate trait implementations. Zero overhead — it's compile-time code generation, not reflection.

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SchemaNode { ... }
🔗 Match It
Match each derive macro to what it provides:
Derive
What you get
All matched! Debug → {:?} formatting. Clone → .clone(). PartialEq → == operator. These are equivalent to implementing ToString(), ICloneable, and IEquatable<T> in C#.

4. Serde Attributes — JSON Serialization

Open settings_manager.rs. You'll see serde annotations that work just like System.Text.Json attributes:

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]       // ← converts snake_case to camelCase in JSON
pub struct ApinoxConfig {
    pub version: i32,
    #[serde(skip_serializing_if = "Option::is_none")]  // ← omit from JSON when None
    pub network: Option<NetworkConfig>,
    pub active_environment: Option<String>,  // → "activeEnvironment" in JSON
    pub open_projects: Option<Vec<String>>,  // → "openProjects" in JSON
}
✍️ Fill in the Blank
To make all fields serialize as camelCase in JSON (e.g., min_occurs → "minOccurs"), what value goes in the serde attribute?
#[serde(rename_all = "")]
camelCase — #[serde(rename_all = "camelCase")]. Other options include "snake_case", "PascalCase", "SCREAMING_SNAKE_CASE". This saves you from adding #[serde(rename = "...")] to every single field.

Why rename = "type"?

#[serde(rename = "type")]
pub node_type: String,

type is a reserved keyword in Rust — you can't name a field type. So the Rust field is node_type but serde maps it to "type" in JSON automatically.

5. Enums — Much More Powerful Than C#

In C#, an enum is a named integer. In Rust, each variant can hold different data. This is called an algebraic data type.

From testing/test_runner.rs:

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "lowercase")]
pub enum TestStepType {
    Soap {
        operation: ServiceOperation,
        soap_version: String,
        values: HashMap<String, String>,
        endpoint: Option<String>,
    },
    Http {
        method: String,
        url: String,
        headers: HashMap<String, String>,
        body: Option<String>,
    },
    Delay {
        ms: u64,
    },
}
✅ True or False
Rust enums can hold different data in each variant (like a union type).
True! Rust enums are algebraic data types — each variant can carry completely different fields. Soap carries an operation and headers, Delay just carries a millisecond count. This is impossible with C# enums (you'd need a class hierarchy). The JSON representation uses a "type" discriminator field to tell them apart.

Using enums: exhaustive match

match &step.step_type {
    TestStepType::Soap { operation, soap_version, .. } => {
        // ".." ignores remaining fields
        execute_soap(operation, soap_version).await?;
    }
    TestStepType::Http { method, url, .. } => {
        execute_http(method, url).await?;
    }
    TestStepType::Delay { ms } => {
        tokio::time::sleep(Duration::from_millis(*ms)).await;
    }
}
The exhaustive match rule: If you forget to handle a variant, Rust won't compile. No silent runtime crashes. Add a new enum variant? Every match in the codebase that doesn't have a _ => wildcard arm will fail to compile, pointing you to every place you need to update.

6. The #[serde(tag = "type")] Pattern

This attribute makes serde include a discriminator field in the JSON output:

{ "type": "soap", "operation": {...}, "soap_version": "1.1" }
{ "type": "http", "method": "GET", "url": "https://..." }
{ "type": "delay", "ms": 1000 }
🤔 Predict the Output
What JSON key does the Rust field original_endpoint become when the struct has #[serde(rename_all = "camelCase")]?
"originalEndpoint" — serde converts snake_case field names to camelCase automatically. Each _ followed by a lowercase letter becomes an uppercase letter. No per-field annotation needed.
