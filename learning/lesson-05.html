1. Rust async/await — Familiar Syntax, Different Execution

The syntax looks just like C# and JavaScript. The big difference is that Rust's async is zero-cost and requires a runtime — futures don't execute at all until they're polled by an executor.

// Rust
pub async fn execute(&self, request: SoapRequest) -> Result<SoapResponse> {
    let response = self.http_client
        .post(&endpoint)
        .send()
        .await?;       // await suspends here until the HTTP response arrives
    Ok(response)
}
// C# — almost identical syntax
public async Task<SoapResponse> ExecuteAsync(SoapRequest request) {
    var response = await _httpClient.PostAsync(endpoint, content);
    return response;
}

Key difference: In C#, calling an async method starts it immediately. In Rust, calling an async function returns a Future that does nothing until .awaited or spawned onto an executor. It's like having all your async methods be lazy by default.
✍️ Fill in the Blank
Complete the call to suspend execution until the async operation completes:
let response = self.http_client.post(&url).send().?;
await — note the suffix position in Rust: .await comes after the expression, not before it like in C#. This makes chaining much cleaner: client.post(url).send().await?.json().await?. The ? after handles any error from the send.

2. Tokio — The Async Runtime

Tauri uses Tokio under the hood. You'll see it in Cargo.toml and lib.rs:

[dependencies]
tokio = { version = "1", features = ["full"] }
// lib.rs — Tauri bootstraps a Tokio runtime automatically.
// In standalone Rust apps, you'd use:
#[tokio::main]
async fn main() {
    // Now you can .await here
}

Tokio provides:

  • A thread pool to execute async tasks
  • tokio::spawn() — run a Future in the background (like Task.Run)
  • tokio::time::sleep() — async sleep
  • tokio::sync::Mutex — async-safe locking
🔘 Multiple Choice
What happens if you call an async fn in Rust without .await?
✓ A dormant Future! Rust will even warn: "unused Future that must be used". Futures are lazy — nothing happens until you .await them (or spawn them with tokio::spawn()). This is unlike C# where calling an async method starts it immediately, whether you await it or not.

3. Generics — Parameterising Over Types

Rust generics look like C# generics but with more expressive constraints. From testing/test_runner.rs:

// T must implement the Serialize trait
async fn send_request<T: Serialize>(body: T) -> Result<Response> { ... }

// Multiple bounds with +
async fn process<T: Serialize + Debug + Clone>(item: T) { ... }

// Where clause — same thing, cleaner syntax for complex bounds
async fn process<T>(item: T) -> Result<()>
where
    T: Serialize + Debug + Clone,
{ ... }
// C# generics with constraints
async Task Process<T>(T item)
    where T : ISerializable, ICloneable
{ ... }

✅ True or False
An async fn in Rust must always return a Result<T, E>.
False! An async fn can return anything: async fn greet() -> String, async fn compute() -> u32, or even async fn fire_and_forget() { } (returns ()). The async keyword just means the function returns a Future — the actual output type is whatever you declare after ->.

4. Tauri Commands — Rust Functions Called from JavaScript

The #[tauri::command] attribute exposes a Rust function to the Tauri webview. These are the RPC boundary between your React frontend and Rust backend. Look at soap/commands.rs:

#[tauri::command]
pub async fn execute_soap_request(
    request: SoapRequest,       // auto-deserialized from JS object
    app_handle: AppHandle,      // Tauri injects this — no JS needed
) -> Result<SoapResponse, String>
//                         ↑ must serialize to JS — String works, custom types work
{
    let client = SoapClient::new();
    client.execute(request).await
        .map_err(|e| e.to_string())
}

Register commands in lib.rs tauri::Builder:

tauri::Builder::default()
    .invoke_handler(tauri::generate_handler![
        execute_soap_request,
        save_project,
        load_project,
    ])
    .run(tauri::generate_context!())?;

Call from JavaScript:

import { invoke } from '@tauri-apps/api';

const response = await invoke('execute_soap_request', {
    request: { endpoint, body, headers }
});

✍️ Fill in the Blank
What attribute exposes a Rust function to Tauri's JavaScript bridge? Fill in just the crate name:
#[::command]
tauri — the full attribute is #[tauri::command]. Tauri uses this to generate the serialization/deserialization glue code automatically. Arguments from JS are deserialized from JSON, and the return value is serialized back to JSON. The AppHandle and State parameters are special — Tauri injects them automatically without the caller needing to pass them.

5. Conditional Compilation with #[cfg(...)]

You'll see this throughout the codebase for platform-specific code:

// Only compile on Windows
#[cfg(target_os = "windows")]
fn get_windows_cert_store() -> CertStore { ... }

// Only compile on macOS or Linux
#[cfg(not(target_os = "windows"))]
fn get_unix_keyring() -> Keyring { ... }

// Include code only when a feature flag is enabled:
#[cfg(feature = "dangerous-test-mode")]
pub fn bypass_auth() { ... }

// Different imports per platform
#[cfg(target_os = "macos")]
use security_framework::passwords::*;

In secret_storage.rs:

#[cfg(target_os = "linux")]
fn store_secret_linux(key: &str, secret: &str) -> Result<()> {
    // Linux secret-service implementation
}

#[cfg(not(target_os = "linux"))]
fn store_secret_keychain(key: &str, secret: &str) -> Result<()> {
    // macOS/Windows implementation
}
