1. Box<T> — Single Owner on the Heap

A Box allocates its value on the heap and gives you one owner of it:

// Most values live on the stack — Box puts them on the heap:
let x: Box<i32> = Box::new(42);
println!("{}", *x);   // deref to get the value

// Main use case 1 — recursive types (size unknown at compile time):
enum Tree {
    Leaf(i32),
    Node(Box<Tree>, Box<Tree>),  // without Box this is infinite size
}

// Main use case 2 — large data you don't want to copy:
let big_data = Box::new([0u8; 1_000_000]);  // 1MB on heap, not stack

// Main use case 3 — trait objects (dyn Trait):
let handlers: Vec<Box<dyn Fn()>> = vec![
    Box::new(|| println!("handler 1")),
    Box::new(|| println!("handler 2")),
];
C# analogy: In C#, all objects are on the heap by default. In Rust, values are on the stack by default. Box is the explicit "put this on the heap" wrapper.
Enterprise thread-safety theater: the architect mandates that all shared state go through an Event Bus "to decouple the components". The Event Bus is a synchronized singleton. The singleton holds a HashMap of listener lists. The listeners are called on the emitter's thread. You've rebuilt a worse version of what you started with, but now it has a diagram.

2. Rc<T> and Arc<T> — Reference Counting

Multiple owners of the same data — without a garbage collector:

use std::rc::Rc;          // single-threaded
use std::sync::Arc;       // multi-threaded (Atomic reference count)

// Arc — sharing config across async tasks in APInox-style code:
let config = Arc::new(AppConfig::load()?);

let c1 = Arc::clone(&config);   // increment ref count — cheap
let c2 = Arc::clone(&config);   // increment again

tokio::spawn(async move {
    println!("{}", c1.server_url);  // c1 dropped here → ref count -1
});
tokio::spawn(async move {
    println!("{}", c2.server_url);  // c2 dropped here → ref count -1
});
// config dropped here → ref count -1 → if 0, memory freed

✅ True or False
Arc::clone(&value) makes a deep copy of the data inside the Arc.
False! Arc::clone just increments the atomic reference counter — it creates a new pointer to the same data. No copying of T occurs. This is why Arc is used for sharing large data structures across threads — you pass cheap pointer copies, not copies of the data. This is exactly what happens with C# object references, but made explicit.

3. Mutex<T> — Shared Mutable State

Arc<T> alone is read-only — you need Mutex for shared mutable state:

Arc lets multiple threads share ownership; Mutex ensures only one has write access at a time
use std::sync::{Arc, Mutex};

// The classic pattern — shared mutable counter across threads:
let counter = Arc::new(Mutex::new(0u32));

let c1 = Arc::clone(&counter);
tokio::spawn(async move {
    let mut guard = c1.lock().await;  // blocks until lock acquired
    *guard += 1;
    // guard dropped here → lock released automatically
});

// From workflow/engine.rs — shared results accumulator:
let results = Arc::new(Mutex::new(Vec::new()));
let r = Arc::clone(&results);
tokio::spawn(async move {
    let mut vec = r.lock().await;
    vec.push(step_result);
});

// Read the accumulated results:
let final_results = results.lock().await.clone();
Deadlock prevention: Keep locks as short-lived as possible. Never hold two locks at once. Tokio's async Mutex (tokio::sync::Mutex) doesn't block the thread — it yields to other tasks instead.
Salesforce's concurrency model: you get one thread. One. Per transaction. Apex is strictly single-threaded. Want parallelism? Queueable Apex — 50 jobs per transaction. Future methods — fire and forget, good luck. Batch Apex — 5 million records, 200 at a time, over 2 hours, might timeout. Platform Events — async, eventually consistent, "eventually" being the operative word. This is called "the Salesforce execution model".
🐛 Spot the Bug
This code will deadlock. Click the problematic line:
let m = Arc::new(Mutex::new(0)); let guard1 = m.lock().unwrap(); let guard2 = m.lock().unwrap(); println!("{} {}", *guard1, *guard2);
Line 3! guard1 holds the lock. Line 3 tries to acquire the same lock — but guard1 hasn't been dropped yet (it's still in scope). The thread blocks forever waiting for itself to release the lock. Fix: use a block scope to drop guard1 before locking again, or use RwLock if you need multiple concurrent readers.
Thread safety was marked 'won't fix' for three sprints because 'it only happens under load and we don't have load'. Sprint 12: production goes down. It was the race condition. The post-mortem action item: 'add thread safety'. Story points: 13. It was the Arc<Mutex<T>> pattern. One line change. The 13 points were for 'investigation and risk assessment'. The risk was: not doing it sooner.

4. RwLock<T> — Many Readers, One Writer

use tokio::sync::RwLock;

// Great for shared config — read often, write rarely:
let config = Arc::new(RwLock::new(AppConfig::default()));

// Multiple concurrent readers — no blocking each other:
let reader1 = Arc::clone(&config);
let reader2 = Arc::clone(&config);

tokio::spawn(async move {
    let cfg = reader1.read().await;  // read lock — shared with others
    println!("{}", cfg.endpoint);
});

tokio::spawn(async move {
    let cfg = reader2.read().await;  // both can hold read locks simultaneously
    println!("{}", cfg.timeout);
});

// Exclusive writer — blocks until all readers are done:
let writer = Arc::clone(&config);
tokio::spawn(async move {
    let mut cfg = writer.write().await;  // exclusive lock
    cfg.endpoint = "https://new-url.com".to_string();
});
🔘 Multiple Choice
When should you prefer RwLock over Mutex?
✓ Read-heavy workloads! RwLock allows multiple concurrent readers. If reads are 95% of operations (like reading config or a cache), RwLock significantly reduces contention. If writes are frequent, the overhead of RwLock's tracking logic can make it slower than a simple Mutex. For write-heavy or equal read/write patterns, Mutex is often better.
Oracle's concurrency solution is row-level locking. Fine. Until you SELECT FOR UPDATE and forget to commit. Or two processes lock rows in opposite order. You get a deadlock. Oracle detects it and kills one session — which one is "implementation-defined". Siebel adds a workflow layer on top of this that locks rows you didn't know existed. Production deadlocks every Friday at 5pm. The DBA blames the devs. The devs blame Siebel. Siebel blames Oracle. Oracle sends an invoice.

5. RefCell<T> — Interior Mutability (Single Thread)

Sometimes you need to mutate data through a shared reference. RefCell moves borrow checking to runtime:

use std::cell::RefCell;

// Normally you can't mutate through a shared reference:
let data = Rc::new(vec![1, 2, 3]);
// data.borrow_mut().push(4);  // ERROR with Rc alone

// RefCell enables it — borrow checked at RUNTIME (not compile time):
let data = Rc::new(RefCell::new(vec![1, 2, 3]));

let d1 = Rc::clone(&data);
let d2 = Rc::clone(&data);

d1.borrow_mut().push(4);   // runtime borrow check — OK
d2.borrow().len();         // read — OK (after d1's borrow released)

// PANICS at runtime if you try two mutable borrows:
// let _a = data.borrow_mut();
// let _b = data.borrow_mut();  // panic! already mutably borrowed
