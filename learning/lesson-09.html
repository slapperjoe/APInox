1. Modules — Rust's Namespace System

Rust doesn't use folders/namespaces the way C# does. The module system is explicit — you declare what exists. From lib.rs:

// lib.rs — the crate root declares all submodules:
mod project_storage;     // private — only this crate can use it
mod secret_storage;      // private
pub mod settings_manager; // public — external code can access

pub mod soap;            // exposes the entire soap/ subdirectory
pub mod testing;
pub mod workflow;
pub mod parsers;

Each mod foo; declaration tells the compiler to look for either:

  • src/foo.rs — a single file module
  • src/foo/mod.rs — a directory module
// src/soap/mod.rs — the soap module's root:
pub mod client;     // makes soap/client.rs accessible as soap::client
pub mod commands;
pub mod envelope_builder;
pub mod ws_security;

// Re-export commonly used items to the module surface:
pub use client::SoapClient;
pub use commands::execute_soap_request;
✅ True or False
In Rust, if you create a file src/database.rs, it is automatically included in your project.
False! Rust's module system is explicit. You must declare mod database; in lib.rs (or the parent module) for the file to be compiled. Files that aren't declared are completely ignored by the compiler. This is intentional — it prevents accidental inclusion of abandoned files and makes the module tree visible in one place.

2. Visibility Modifiers

ModifierAccessible fromC# equivalent
(none)Only within the current moduleprivate
pubAnywherepublic
pub(crate)Anywhere within this crateinternal

mod authentication {
    pub struct User {
        pub username: String,           // pub — accessible everywhere
        pub(crate) password_hash: String, // internal — only this crate
        session_token: String,           // private — only this module
    }

    pub fn login(username: &str) -> User { ... }  // public
    fn hash_password(pw: &str) -> String { ... }   // private helper
    pub(crate) fn validate_session(token: &str) { ... } // crate-internal
}
🔘 Multiple Choice
Your Tauri app has a function that should be callable from any module in your crate, but not exposed to JavaScript via invoke(). What visibility do you use?
✓ pub(crate)! Tauri commands must be pub to be registered with tauri::generate_handler!. But for internal utility functions you want shared across modules — not exposed to the JS layer — pub(crate) is the right choice. It's Rust's equivalent of C#'s internal: visible within the assembly (crate) but not externally.

3. use — Bringing Items into Scope

// Bring specific items into scope:
use std::collections::HashMap;
use std::path::PathBuf;
use std::fs;

// Multiple items from one module:
use std::io::{BufRead, BufReader, Write};

// Rename on import:
use std::collections::HashMap as Map;
use serde_json::Value as JsonValue;    // seen in assertion_runner.rs!

// Glob import (use sparingly — makes it hard to trace where things come from):
use std::prelude::*;

// Re-export — expose an item as if it's part of your module:
pub use crate::soap::SoapClient;   // callers can use `your_crate::SoapClient`

// Relative paths:
use super::utils::format_xml;   // go up one module level
use self::models::User;         // explicitly current module
use crate::settings_manager;    // from the crate root
✍️ Fill in the Blank
Bring HashMap and HashSet into scope from std::collections in one line:
use std::collections::{};
HashMap, HashSet (order doesn't matter) — the { } syntax lets you import multiple items from one path. This is called a "use tree" and you can nest them: use std::{collections::HashMap, path::PathBuf, fs}; — all from one use statement.

4. The APInox Module Tree

Here's how the codebase is structured — notice everything flows from lib.rs:

// lib.rs (crate root)
├── mod project_storage      → src/project_storage.rs
├── mod secret_storage       → src/secret_storage.rs
├── pub mod settings_manager → src/settings_manager.rs
├── pub mod soap             → src/soap/mod.rs
│   ├── pub mod client       → src/soap/client.rs
│   ├── pub mod commands     → src/soap/commands.rs
│   └── pub mod ws_security  → src/soap/ws_security.rs
├── pub mod testing          → src/testing/mod.rs
│   ├── pub mod test_runner  → src/testing/test_runner.rs
│   └── pub mod assertion_runner → src/testing/assertion_runner.rs
└── pub mod workflow         → src/workflow/mod.rs
    └── pub mod engine       → src/workflow/engine.rs

Accessing items across modules:

// In soap/commands.rs, access the testing module:
use crate::testing::TestRunner;
//   ↑ always start from crate root for absolute paths

// OR use super for relative:
use super::client::SoapClient;  // go up to soap/, then into client
🐛 Spot the Bug
Code in soap/commands.rs tries to use a private function from project_storage.rs. Click the problem:
// In project_storage.rs: fn load_from_disk(path: &str) -> Result<Project> { ... } // In soap/commands.rs: use crate::project_storage::load_from_disk;
Line 2! load_from_disk has no pub — it's private to project_storage. Even though it's accessible within that module, no other module can import it. Fix: add pub fn if it's meant to be shared, or pub(crate) fn to limit it to this crate. Private functions are a form of encapsulation — they protect internal implementation details.

5. Inline Modules & Test Modules

You can also declare modules inline, without creating a file. This is especially useful for test code:

// Inline module — all in the same file:
mod utils {
    pub fn format_xml(s: &str) -> String {
        // implementation
    }
}

// The #[cfg(test)] module pattern — standard across ALL Rust codebases:
// src/my_module.rs
pub fn add(a: i32, b: i32) -> i32 { a + b }

#[cfg(test)]         // ← only compiled when running tests
mod tests {
    use super::*;    // ← import everything from the parent module

    #[test]
    fn test_add() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn test_add_negative() {
        assert_eq!(add(-1, 1), 0);
    }
}
🤔 Predict the Output
What does use super::* do inside a #[cfg(test)] mod tests block?
It imports everything from the parent module — all functions, structs, and types defined in the file that contains the test module. Since the test module is a child of the file's module, super refers to that parent. This gives tests access to private functions (not just pub ones) — because parent-child module relationships allow access to private items. It's intentional: tests should be able to test internals.
