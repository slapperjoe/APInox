1. The Plugin Architecture

Tauri v2 ships OS integrations as optional plugins rather than bundling everything. You only include what you need, keeping binary size small. In APInox's lib.rs:

tauri::Builder::default()
    .plugin(tauri_plugin_process::init())       // app.exit(), restart()
    .plugin(tauri_plugin_dialog::init())         // file open/save dialogs
    .plugin(tauri_plugin_clipboard_manager::init()) // copy/paste
    .plugin(tauri_plugin_store::Builder::default().build()) // key/value store
    .plugin(tauri_plugin_opener::init())         // open URLs in browser
    .plugin(tauri_plugin_shell::init())          // run shell commands
    // ...
    .invoke_handler(tauri::generate_handler![...])

Each plugin:

  1. Adds Rust-side capabilities
  2. Exposes TypeScript bindings via @tauri-apps/plugin-X NPM packages
  3. Requires permissions declared in capabilities/ files (covered in Lesson 21)
# Cargo.toml — Rust side of plugins
[dependencies]
tauri-plugin-dialog = "2"
tauri-plugin-clipboard-manager = "2"
tauri-plugin-shell = "2"
tauri-plugin-opener = "2"
tauri-plugin-store = "2"
// package.json — TypeScript side (in webview)
{
  "dependencies": {
    "@tauri-apps/api": "^2.0.0",
    "@tauri-apps/plugin-dialog": "^2.0.0",
    "@tauri-apps/plugin-clipboard-manager": "^2.0.0",
    "@tauri-apps/plugin-shell": "^2.0.0",
    "@tauri-apps/plugin-store": "^2.0.0"
  }
}
✅ True or False
In Tauri v2, adding a plugin like tauri_plugin_dialog in Rust is enough — you don't need to add anything to Cargo.toml or the NPM packages.
False! You need three things: (1) The Rust crate in Cargo.toml (tauri-plugin-dialog = "2"), (2) .plugin(tauri_plugin_dialog::init()) in your builder chain, and (3) the NPM package in your webview's package.json (@tauri-apps/plugin-dialog). Plus permissions in the capabilities file (Lesson 21). It's a bit verbose, but it's explicit — you can't accidentally ship features you didn't intend to include.

2. File Dialogs — Open & Save

The dialog plugin provides native OS file pickers. APInox uses these to open/save project folders:

import { open, save } from '@tauri-apps/plugin-dialog';

// Open a folder picker
const selectedPath = await open({
    directory: true,        // folder picker, not file
    multiple: false,
    title: 'Select project folder',
});
// selectedPath: string | null (null if cancelled)

// Open a file picker with filters
const filePath = await open({
    multiple: false,
    filters: [
        { name: 'WSDL Files', extensions: ['wsdl', 'xml'] },
        { name: 'All Files', extensions: ['*'] },
    ],
    title: 'Open WSDL file',
});

// Save dialog
const savePath = await save({
    defaultPath: '~/Documents/my-project',
    filters: [{ name: 'APInox Project', extensions: ['apinox'] }],
});
Unlike a browser's <input type="file">, Tauri's dialog returns the actual file path on disk — not a sandboxed File object. You can then pass this path to a Rust command that reads/writes the file directly.
🔘 Multiple Choice
A user picks a file in Tauri's open dialog and the path "/Users/mark/wsdl/service.wsdl" is returned. What's the best next step?
Pass to Rust via invoke! The webview is a sandboxed browser — it can't read arbitrary file paths directly (no Node.js, no filesystem access by default). The pattern is: dialog → path string → invoke('my_command', { path }) → Rust reads the file → returns data back to TypeScript. Rust has unrestricted OS access; the webview goes through Rust for anything system-level.

3. Shell Plugin — Opening URLs & Running Commands

// @tauri-apps/plugin-opener — open URLs in the default browser:
import { openUrl } from '@tauri-apps/plugin-opener';
await openUrl('https://docs.apinox.dev');

// @tauri-apps/plugin-shell — run shell commands (needs allowlist!):
import { Command } from '@tauri-apps/plugin-shell';
const output = await Command.create('curl', ['-s', 'https://api.example.com'])
    .execute();
console.log(output.stdout);  // response body
// capabilities/default.json — must explicitly allow shell commands:
{
  "permissions": [
    "opener:default",
    {
      "identifier": "opener:allow-open-path",
      "allow": [{ "path": "$TEMP/**" }]
    }
  ]
}
⚠️ The shell plugin is powerful and dangerous — it runs arbitrary OS commands. In Tauri v2, you must whitelist each allowed command in the capabilities config. Never expose user-controlled input to shell commands (injection risk). APInox uses the opener plugin for URLs and the dialog plugin for files — not raw shell access.

4. Clipboard Manager

import { writeText, readText } from '@tauri-apps/plugin-clipboard-manager';

// Copy to clipboard
await writeText('SOAPAction: "http://example.com/GetUser"');

// Read from clipboard
const text = await readText();
if (text) {
    pasteIntoRequestEditor(text);
}

APInox uses this to let users copy request XML or SOAP endpoints. The plugin handles OS-level clipboard access cleanly — on macOS it requests the NSPasteboardUsage permission automatically.

5. Plugin Store — Persistent Key-Value Storage

The store plugin provides a simple JSON-backed key-value store — useful for persisting UI state, preferences, and window position between sessions:

import { load } from '@tauri-apps/plugin-store';

// Load (or create) a store file
const store = await load('settings.json', { autoSave: true });

// Write values
await store.set('theme', 'dark');
await store.set('lastProject', '/Users/mark/Projects/APInox');
await store.set('windowBounds', { x: 100, y: 200, width: 1400, height: 900 });

// Read values
const theme = await store.get<string>('theme');
const lastProject = await store.get<string>('lastProject');

// Delete
await store.delete('temporaryKey');
The store file lives in the app's data directory (~/.local/share/com.apinox.dev/ on Linux, ~/Library/Application Support/app.apinox.dev1/ on macOS). For more complex persistence (structured project data), APInox uses the FolderProjectStorage Rust module — the store plugin is better for lightweight settings.
🔗 Match It
Match the Tauri plugin to its primary use case:
All matched! Tauri's plugin separation keeps each dependency optional. A simple note-taking app might only need the store plugin. A SOAP API tester like APInox needs dialog (project folders), store (recent files), opener (doc links), and clipboard (copy endpoints/XML). None of these require custom Rust code — they're pre-built integrations.

6. App Directories — Where to Store Data

// In Rust commands — access standard OS directories:
use tauri::Manager;

#[tauri::command]
pub fn get_config_dir(app: tauri::AppHandle) -> Result<String, String> {
    let path = app.path().app_config_dir()
        .map_err(|e| e.to_string())?;
    Ok(path.to_string_lossy().to_string())
}

// Available path types:
// app.path().app_config_dir()   → ~/.config/app.apinox.dev1/
// app.path().app_data_dir()     → ~/.local/share/app.apinox.dev1/
// app.path().app_log_dir()      → ~/.local/share/app.apinox.dev1/logs/
// app.path().home_dir()         → /Users/mark/
// app.path().download_dir()     → /Users/mark/Downloads/

APInox stores its config in ~/.apinox/ — a custom path, not the standard app config dir. In lib.rs:

// lib.rs — APInox uses its own ~/.apinox/ folder:
static CONFIG_DIR: Mutex<Option<String>> = Mutex::new(None);

fn get_apinox_config_dir() -> PathBuf {
    let home = dirs::home_dir().unwrap_or_default();
    home.join(".apinox")
}
✍️ Fill in the Blank
To access standard OS paths in a Rust command, which trait must be in scope?
use tauri::;

#[tauri::command]
pub fn get_data_dir(app: tauri::AppHandle) -> String {
    app.path().app_data_dir().unwrap().to_string_lossy().to_string()
}
Manager — tauri::Manager is a trait that adds the .path() method (and others like .get_webview_window(), .manage(), .emit()) to AppHandle, App, and window types. Without use tauri::Manager; in scope, calling app.path() gives a "method not found" error.

7. Writing Your Own Plugin (Brief Overview)

If an official plugin doesn't cover your need, you can write custom plugins. For APInox, most custom functionality is implemented as plain #[tauri::command] functions rather than full plugins — plugins are better when you want to package reusable functionality for multiple apps:

// Skeleton of a custom plugin
pub fn init<R: tauri::Runtime>() -> tauri::plugin::TauriPlugin<R> {
    tauri::plugin::Builder::new("my-plugin")
        .invoke_handler(tauri::generate_handler![
            my_plugin_command_one,
            my_plugin_command_two,
        ])
        .setup(|app, _api| {
            // Plugin initialization
            Ok(())
        })
        .build()
}

// Then in main app:
.plugin(my_plugin::init())
🔮 Predict & Reveal
APInox calls app.exit(0) in the quit_app command. If instead you called the TypeScript equivalent process.exit() from the webview — what would happen?
It wouldn't work. process is a Node.js global — it doesn't exist in the webview's browser environment (Tauri's webview is not a Node.js environment). You'd get a ReferenceError: process is not defined. To quit the app from the webview you must call invoke('quit_app') which runs the Rust app.exit(0). Tauri maintains a strict separation: the webview is a sandboxed browser, the Rust layer has OS access.
