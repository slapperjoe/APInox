Serde & Serialization

⏱ ~25 min 📁 parsers/wsdl/types.rs, testing/assertion_runner.rs, workflow/engine.rs

0 of 5 questions answered

Serde (Serialise/Deserialise) is the de-facto standard for JSON in Rust. APInox uses it extensively — the Tauri command bridge relies on it entirely for passing data between Rust and JavaScript.
XML was designed in 1996 to be a "simple" data format. It now has: namespaces, schema (XSD), validation (Schematron), transformation (XSLT), querying (XPath), querying with updates (XQuery), signatures (XML-DSig), encryption (XML-Enc), and a 900-page specification. SOAP adds a mandatory envelope, a body, optional headers, and a WSDL that may or may not agree with the actual service. APInox exists because someone had to interact with all of this in 2024. They wept. Then they wrote a GUI for it.

1. The Basics — derive Macros

use serde::{Deserialize, Serialize};

// That's all you need for 90% of cases:
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SchemaNode {
    pub name: String,
    pub kind: String,
    pub children: Option<Vec<SchemaNode>>,
}

// Serialise to JSON:
let node = SchemaNode { name: "Root".to_string(), kind: "complex".to_string(), children: None };
let json = serde_json::to_string(&node)?;
// {"name":"Root","kind":"complex","children":null}

let pretty = serde_json::to_string_pretty(&node)?;
// {
//   "name": "Root",
//   "kind": "complex",
//   "children": null
// }

// Deserialise from JSON:
let json_str = r#"{"name":"Root","kind":"complex","children":null}"#;
let node: SchemaNode = serde_json::from_str(json_str)?;
// C# equivalent with System.Text.Json:
[JsonSerializable(typeof(SchemaNode))]
public class SchemaNode {
    public string Name { get; set; }
    public string Kind { get; set; }
    public List<SchemaNode>? Children { get; set; }
}
var json = JsonSerializer.Serialize(node);
var node = JsonSerializer.Deserialize<SchemaNode>(json);
✅ True or False
Serde's #[derive(Serialize, Deserialize)] works on enums as well as structs.
True! Serde handles enums extremely well — far better than most other languages. From assertion_runner.rs: #[derive(Serialize, Deserialize)] pub enum AssertionType { XPath { xpath: String, expected: String }, StatusCode { code: u16 } }. By default, enum variants are serialised with a type tag. This is controlled with #[serde(tag = "type")].

2. Essential Serde Attributes — Real Examples from the Codebase

From parsers/wsdl/types.rs:

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SchemaNode {
    pub name: String,

    #[serde(rename = "type")]
    pub node_type: String,
    //   ↑ field is called node_type in Rust (reserved keyword clash)
    //     but serialises as "type" in JSON

    #[serde(skip_serializing_if = "Option::is_none")]
    pub min_occurs: Option<String>,
    //   ↑ omit from JSON entirely if None (not "min_occurs":null)

    #[serde(skip_serializing_if = "Option::is_none")]
    pub children: Option<Vec<SchemaNode>>,
}

// rename_all — applies a naming convention to ALL fields:
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct ServiceOperation {
    pub target_namespace: Option<String>,   // → "targetNamespace" in JSON
    pub original_endpoint: Option<String>,  // → "originalEndpoint" in JSON
    pub port_name: Option<String>,          // → "portName" in JSON
}
AttributeEffectC# equivalent
#[serde(rename = "foo")]Use "foo" in JSON[JsonPropertyName("foo")]
#[serde(rename_all = "camelCase")]Convert all fields to camelCaseJsonNamingPolicy.CamelCase
#[serde(skip_serializing_if = "Option::is_none")]Omit field if None[JsonIgnore(Condition = WhenWritingNull)]
#[serde(default)]Use Default if field missing from JSON[JsonIgnore(Condition = WhenWritingDefault)]
#[serde(skip)]Never serialise this field[JsonIgnore]
#[serde(flatten)]Inline struct fields into parent(no direct equivalent)
#[serde(tag = "type")]Enum uses "type" field for variant[JsonDerivedType] (polymorphism)
✍️ Fill in the Blank
What attribute makes all fields of this struct serialise as camelCase JSON keys?
#[derive(Serialize, Deserialize)]
#[serde()]
pub struct Config {
    pub server_url: String, // → "serverUrl"
    pub max_retries: u32, // → "maxRetries"
}
rename_all = "camelCase" — applied at the struct level, it converts snake_case Rust field names to camelCase for JSON. Options include: "camelCase", "PascalCase", "snake_case", "SCREAMING_SNAKE_CASE", "kebab-case". APInox uses camelCase on ServiceOperation so its JSON matches JavaScript naming conventions.
SAP's data interchange format is IDoc — Intermediate Document. It's fixed-width, segment-based, and dates to the early 1990s. Each IDoc type has segments, each segment has fields, each field is padded to a fixed length. The MATNR field (material number) is 18 characters, left-padded with zeros. You will spend an entire day writing a parser for the leading zeros. Then you'll find the SAP BAPI that does it wrong anyway. There's also ALE, EDI, IDEX, and XI/PI/PO — all different ways to say "we're sending you an IDoc". Sorry.

3. Enum Serialisation with Tags

From testing/assertion_runner.rs:

#[derive(Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "lowercase")]
pub enum AssertionType {
    XPath {
        xpath: String,
        expected: String,
    },
    StatusCode {
        code: u16,
    },
}

This serialises to JSON like:

{ "type": "xpath", "xpath": "//Name", "expected": "Alice" }
{ "type": "statuscode", "code": 200 }

Without #[serde(tag)], serde wraps in variant name:

{ "XPath": { "xpath": "//Name", "expected": "Alice" } }
🔘 Multiple Choice
A None field with #[serde(skip_serializing_if = "Option::is_none")] serialises as:
✓ Omitted entirely! This is important for clean API responses — you don't litter the JSON with dozens of null fields. APInox's SchemaNode has 6 optional fields that are only included when present. Without skip_serializing_if, None serialises as null in JSON. With it, the key disappears. When deserialising, a missing field becomes None automatically.

4. serde_json::Value — Untyped JSON

Sometimes you need to work with arbitrary JSON you don't know the shape of:

use serde_json::Value;

// Parse any JSON without knowing the structure:
let raw: Value = serde_json::from_str(json_str)?;

// Navigate with indexing (returns &Value or Value::Null if missing):
let name = &raw["user"]["name"];
let first_tag = &raw["tags"][0];

// Pattern match on the type:
match &raw {
    Value::Object(map) => println!("{} keys", map.len()),
    Value::Array(arr)  => println!("{} items", arr.len()),
    Value::String(s)   => println!("String: {}", s),
    Value::Number(n)   => println!("Number: {}", n),
    Value::Bool(b)     => println!("Bool: {}", b),
    Value::Null        => println!("null"),
}

// Build JSON dynamically:
use serde_json::json;
let payload = json!({
    "name": "Alice",
    "age": 30,
    "tags": ["rust", "tauri"],
    "active": true
});
🐛 Spot the Bug
This serialisation won't work as expected. Click the problem line:
use serde::Serialize; #[derive(Serialize)] struct Response {     pub data: Vec<std::sync::Mutex<String>>, }
Line 2! std::sync::Mutex doesn't implement Serialize, so #[derive(Serialize)] fails on a struct containing it. Serde can only serialise types that implement Serialize. The fix: don't put a Mutex in a data structure you want to serialise. Serialise the inner value: let data: Vec<String> = mutex_vec.iter().map(|m| m.lock().unwrap().clone()).collect();
Oracle's approach to data serialisation: if it's going to another Oracle system, use database links. If it's going somewhere else, generate XML via DBMS_XMLGEN, wrap it in a SOAP envelope using UTL_HTTP, and POST it manually from PL/SQL. The HTTP client in PL/SQL (UTL_HTTP) requires a wallet for HTTPS. The wallet setup involves a 14-step OSS document, an OWM binary, and a call to your DBA who is on holiday.

5. serde(flatten) — Inline a Nested Struct

#[derive(Serialize, Deserialize)]
struct Metadata {
    created_at: String,
    updated_at: String,
}

#[derive(Serialize, Deserialize)]
struct User {
    name: String,
    email: String,
    #[serde(flatten)]
    meta: Metadata,  // fields merged into the parent JSON object
}

// Serialises as:
// { "name": "Alice", "email": "...", "created_at": "...", "updated_at": "..." }
// NOT:
// { "name": "Alice", "meta": { "created_at": "..." } }

// Also seen in workflow/engine.rs for WorkflowStep:
#[derive(Serialize, Deserialize)]
pub struct WorkflowStep {
    pub name: String,
    #[serde(flatten)]
    pub node: WorkflowNode,   // WorkflowNode's fields merged in
}
🔗 Match It
Match the serde attribute to what it does in JSON:
All matched!
• rename = "type" → key is "type" in JSON (not the Rust field name)
• skip_serializing_if = "Option::is_none" → omit key when None
• flatten → inline child struct's fields into the parent object
• tag = "type" → enum variant identified by {"type":"variantname",...}
Serde handles JSON beautifully. XML is where things get philosophical. SOAP XML has: an envelope, a body, optional headers, namespaces that may or may not match the WSDL, elements that appear in the schema as optional but the server requires, and a Content-Type of text/xml (SOAP 1.1) or application/soap+xml (SOAP 1.2) which the server may or may not enforce. The XSD says the date format is ISO 8601. The server sends DD/MM/YYYY. The XSD is wrong. This is normal.
