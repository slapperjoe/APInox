1. Rust Has No Exceptions

In C# and JavaScript, errors propagate via thrown exceptions caught with try/catch. Rust has no exceptions. Functions that can fail return a Result<T, E> value instead. The compiler forces you to handle it.

Java has checked exceptions — you must declare and handle them. A noble idea. The community's adopted solution after 30 years: catch (Exception e) { /* TODO: handle this */ } or just throws Exception on every method signature until it disappears up the call stack. Rust skipped the whole ceremony and made errors actual return values you can't ignore.
C#Rust
throw new Exception("Failed")return Err("Failed".to_string())
try { } catch (Exception e) { }match result { Ok(v) => ..., Err(e) => ... }
Function might throw — no type indicationFunction returns Result<T,E> — always explicit
Result<T, E> forces you to handle both paths — the compiler won't let you ignore an error
// Result is just this enum:
enum Result<T, E> {
    Ok(T),    // success — contains the value
    Err(E),   // failure — contains the error
}

// A function that can fail:
pub async fn save_project(project: serde_json::Value, dir_path: String)
    -> Result<(), String>
//            ↑   ↑
//  success: ()   error: String message
ABAP error handling: CATCH cx_root INTO lx_error. WRITE: / lx_error->get_text( ). That's the good version. The bad version is a DUMP — a full-screen 1990s-style crash report with a 40-character error code like SYSTEM_NO_ROLL that you paste into OSS Notes and pray. SAP's equivalent of Result<T, E> is a 200-page ABAP exception class hierarchy you didn't write, can't modify, and must document in a Change Request before touching.
✅ True or False
In Rust, a function can silently throw an exception that the caller doesn't know about.
False! Rust has no exceptions. If a function can fail, it must return Result<T, E> — that failure possibility is part of the function's type signature. The caller always knows a function can fail because the return type says so.

2. The ? Operator — Early Return on Error

Writing match for every error gets verbose fast. The ? operator is shorthand for "if this failed, return the error immediately":

// Verbose — using match explicitly:
let dir_result = fs::create_dir_all(&dir);
match dir_result {
    Ok(()) => { /* continue */ },
    Err(e) => return Err(format!("Failed to create dir: {}", e)),
}

// Same thing with ? — much cleaner:
fs::create_dir_all(&dir)
    .map_err(|e| format!("Failed to create dir: {}", e))?;
//                                                      ↑ if Err, return early

From project_storage.rs — notice how ? chains through the whole function:

pub async fn save_project(project: serde_json::Value, dir_path: String)
    -> Result<(), String>
{
    let dir = PathBuf::from(&dir_path);

    fs::create_dir_all(&dir)
        .map_err(|e| format!("Failed to create directory: {}", e))?;

    let props_json = serde_json::to_string_pretty(&props)
        .map_err(|e| format!("Failed to serialize: {}", e))?;

    fs::write(&props_path, props_json)
        .map_err(|e| format!("Failed to write: {}", e))?;

    Ok(())   // ← explicit success return
}
.NET MAUI has a global unhandled exception handler: MauiExceptions.UnhandledException. On Android it catches most things. On iOS it catches some things. On Windows it catches different things. On Mac Catalyst it logs to nowhere and your app silently terminates. The documentation says this is "platform-specific behaviour." ✍️ Fill in the Blank
Add the operator that propagates the error if read_file() fails:
let contents = fs::read_to_string(&path);
? — fs::read_to_string(&path)?. If the read fails (Err), the ? immediately returns that error from the current function. If it succeeds (Ok), it unwraps the value and assigns it to contents. The function must return Result for this to work.

3. map_err — Converting Error Types

? requires compatible error types. map_err transforms one error type into another:

// fs::create_dir_all returns Result<(), std::io::Error>
// But our function returns    Result<(), String>
// map_err bridges the gap:
fs::create_dir_all(&dir)
    .map_err(|e| format!("Failed: {}", e))?;
//  ^-- io::Error → String                ↑ then propagate
C# analogy: It's like catching a specific exception and re-throwing a different one — except it's all done as a method chain with no try/catch syntax.

🐛 Spot the Bug
This code compiles but will crash at runtime. Click the dangerous line:
fn get_first_user(users: &Vec<String>) -> String { users.first().unwrap().clone() }
Line 2! .first() returns Option<&String> — it's None if the vec is empty. .unwrap() on None panics (crashes the program). Safer alternatives:
users.first().cloned().unwrap_or_default() — returns empty string if empty
users.first().map(|s| s.clone()).ok_or("No users")? — propagate as error

5. Option Methods — Chaining Without Unwrapping

You'll see Option chains like this in secret_storage.rs:

let config_dir = std::env::var("APINOX_CONFIG_DIR")
    .ok()                        // Result → Option (discard error kind)
    .and_then(|dir|              // transform if Some, propagate None
        if dir.trim().is_empty() { None }
        else { Some(PathBuf::from(dir)) }
    )
    .or_else(|| {               // fallback if still None
        let home = std::env::var("HOME").ok()?;
        Some(PathBuf::from(home).join(".apinox"))
    })
    .ok_or("Could not determine config directory")?; // Option → Result

🔘 Multiple Choice
What does .ok_or("error message") do when called on an Option<T>?
✓ Converts Option to Result! This is the bridge between the two types. Some(v) → Ok(v), None → Err(your_message). Combined with ?, it lets you write: maybe_value.ok_or("Was missing")? — which reads as "give me the value or return early with this error."
Oracle Forms error handling is an ON-ERROR trigger. One trigger. For all errors. You check ERROR_CODE with a massive IF/ELSIF chain, then call BELL to beep at the user. This shipped in production systems used by actual humans until at least 2023. Some of them are still running. In a hospital. Please don't think about that too hard.
✅ True or False
The ? operator can only be used in functions that return Result<T, E>.
False — almost! ? works in functions returning either Result<T, E> or Option<T>. When used in an Option-returning function, ? on a None returns None from the function. This is great for short-circuiting optional value chains without a ton of if let nesting.
