1. Separating Data from Behaviour

In C#, a class contains both fields and methods in one block. In Rust, data (struct) and behaviour (impl) are written separately. Open soap/client.rs:

// DATA: what a SoapClient IS
pub struct SoapClient {
    http_client: Client,   // private field (no pub)
}

// BEHAVIOUR: what a SoapClient DOES
impl SoapClient {
    pub fn new() -> Self { ... }
    pub fn with_client(client: Client) -> Self { ... }
    pub async fn execute(&self, ...) -> Result<SoapResponse> { ... }
}
// C# combines these in one class block
public class SoapClient {
    private readonly HttpClient _httpClient;

    public SoapClient() { ... }
    public async Task<SoapResponse> Execute(...) { ... }
}
You can have multiple impl blocks for the same type — they merge. This lets you split functionality across files or group by feature area.

2. Constructors — The new() Convention

Rust has no new keyword. The convention is an associated function named new() that returns Self:

impl SoapClient {
    /// Create a new SOAP client
    pub fn new() -> Self {
        Self {
            http_client: Client::new(),
        }
    }

    /// Alternative constructor — with a custom HTTP client
    pub fn with_client(client: Client) -> Self {
        Self {
            http_client: client,
        }
    }
}

// Usage — :: for associated functions, . for methods:
let client = SoapClient::new();
let client = SoapClient::with_client(my_client);
✅ True or False
Rust has a new keyword that you use to create struct instances, just like C#.
False! Rust has no new keyword. new() is just a regular associated function — a convention, not a language feature. You can name constructors anything: with_client(), from_config(), default(). They all just return Self.

3. self, &self, &mut self

The first parameter of a method declares its relationship to the instance:

ParameterMeaningC# equivalent
selfTakes ownership — instance is consumed(unusual in C#)
&selfBorrows immutably — read-onlythis in a normal method
&mut selfBorrows mutably — can modify structthis in a method that modifies state

From testing/test_runner.rs:

impl TestRunner {
    pub fn new() -> Self { ... }                           // no self — associated fn

    pub fn set_variable(&mut self, name: String, value: String) {
        self.variables.insert(name, value);                // needs &mut
    }

    pub fn get_variable(&self, name: &str) -> Option<&String> {
        self.variables.get(name)                           // read-only, &self
    }

    fn replace_variables(&self, text: &str) -> String { ... } // private, read-only

    pub async fn run_test_case(&mut self, ...) -> Result<...> {
        // needs &mut because it modifies variables during execution
    }
}
✍️ Fill in the Blank
This method needs to modify the struct's fields. What goes in the blank?
pub fn set_port(& self, port: u16) {
    self.port = port;
}
mut — the signature is &mut self. Without mut, the compiler rejects any attempt to modify self.port. Start with &self and only add mut when the compiler tells you you need it.

4. Traits — Rust's Interfaces

A trait defines methods a type must implement. Just like a C# interface, but with superpowers:

// Define a trait (like an interface)
trait Greet {
    fn hello(&self) -> String;              // required — must implement

    fn goodbye(&self) -> String {          // default implementation
        format!("Goodbye from {}", self.hello())
    }
}

// Implement the trait for a type
impl Greet for SoapClient {
    fn hello(&self) -> String {
        "Hello from SoapClient".to_string()
    }
    // goodbye() is inherited automatically
}
// C# equivalent
interface IGreet {
    string Hello();
    string Goodbye() => $"Goodbye from {Hello()}";  // default method
}
class SoapClient : IGreet {
    public string Hello() => "Hello from SoapClient";
}

🔘 Multiple Choice
In Rust, can you implement a trait for a type that was defined in a third-party crate?
✓ Yes, with the orphan rule! Either the trait or the type must be local to your crate. You can implement Display (foreign trait) for SoapClient (your type). You CAN'T implement Display (foreign) for Vec<String> (also foreign). This prevents conflicting implementations across crates.

5. The Builder Pattern

Rust doesn't have optional/default parameters like C#. The Builder pattern solves this elegantly. Open soap/ws_security.rs:

The Java equivalent is AbstractSingletonProxyFactoryBean. That's a real class in Spring Framework. The Rust equivalent is WsSecurityConfig::new(). One of these names describes what the thing does. Enterprise architects get very upset when you point this out.
#[derive(Debug, Clone, Default)]
pub struct WsSecurityConfig {
    pub username_token: Option<UsernameToken>,  // None by default
    pub timestamp: Option<Timestamp>,
    pub add_nonce: bool,                         // false by default
}

impl WsSecurityConfig {
    pub fn new() -> Self {
        Self::default()   // all fields are None / false
    }

    // Builder methods: take ownership of self, mutate, return self
    pub fn with_username_token(mut self, token: UsernameToken) -> Self {
        self.username_token = Some(token);
        self   // ownership returned to caller — enables chaining
    }

    pub fn with_timestamp(mut self, ts: Timestamp) -> Self {
        self.timestamp = Some(ts);
        self
    }

    pub fn with_nonce(mut self) -> Self {
        self.add_nonce = true;
        self
    }
}

// Usage — fluent chaining:
let security = WsSecurityConfig::new()
    .with_username_token(UsernameToken::text("admin".into(), "pass".into()))
    .with_default_timestamp()
    .with_nonce();
🤔 Predict the Output
Builder methods take mut self (by value). What would happen if you tried to use the config variable after calling a builder method?
let config = WsSecurityConfig::new();
let with_token = config.with_username_token(token);
// Can we still use `config` here?
println!("{:?}", config);
💥 Compile error! with_username_token takes self by value (ownership), so config is moved into the method. After the call, config is gone — only with_token exists. This is intentional: it prevents you from accidentally using a half-configured object. The builder pattern works by chaining all the calls together before assigning to a final variable.

6. Destructors — The Drop Trait

Rust has no ~Foo() destructor syntax. Instead, implement the Drop trait:

impl Drop for SidecarProcess {
    fn drop(&mut self) {
        // called automatically when value goes out of scope
        self.stop();
        log::info!("Sidecar process stopped");
    }
}

In practice, you rarely write Drop in application code. The compiler automatically frees memory when values go out of scope. You only implement Drop for external resources like OS handles, network connections, or file locks.

🐛 Spot the Bug
This impl block has a mistake. Click the problematic line:
impl Counter { pub fn increment(&self) { self.count += 1; } }
Line 2! The method signature says &self (immutable borrow) but the body tries to modify self.count. To mutate a field, the signature must be &mut self: pub fn increment(&mut self). Rust enforces this at compile time — no accidental mutation through a "read-only" reference.
