Lifetimes

⏱ ~35 min ⚠️ Honest warning: this is the dry one

0 of 5 questions answered

⚠️ Fair warning: Lifetimes are the most conceptually heavy part of Rust. This lesson is intentionally straightforward, not jazzy. You rarely write them in application code (APInox barely uses any) — but you will encounter them in library code, and the compiler will occasionally demand them. Skim it now, return when the compiler yells at you.

1. Why Lifetimes Exist

Lifetimes are Rust's way of proving at compile time that a reference never outlives the data it points to. In C#, the GC handles this at runtime. Rust does it at compile time — no GC needed.

// This is the problem lifetimes prevent:
fn bad() -> &String {
    let s = String::from("hello");
    &s  // ERROR: s is dropped at end of function
        // returning a reference to freed memory — use after free
}
// In C++, this compiles and causes undefined behaviour.
// In Rust, it's a compile error.
// This is fine — the reference lives as long as the value:
fn first_word(s: &str) -> &str {
    &s[..s.find(' ').unwrap_or(s.len())]
    // returned slice borrows FROM s — valid as long as s is valid
}
The rule: A reference can't outlive the data it points to. Lifetimes are how the compiler tracks this.
An architect will look at Box<T> and ask if this is the Repository Pattern. It is not. They will then ask if we should wrap it in a factory. You will explain it's a heap allocation. They will ask if the heap should be its own service. You will go home.

2. Lifetime Elision — Why You Don't Usually Write Them

The compiler can infer lifetimes in most common patterns via three elision rules:

// Rule 1: Each reference parameter gets its own lifetime
fn foo(x: &str, y: &str) -> ...
// compiler sees: fn foo<'a, 'b>(x: &'a str, y: &'b str) -> ...

// Rule 2: If there's exactly one input reference, the output gets that lifetime
fn first_word(s: &str) -> &str
// compiler sees: fn first_word<'a>(s: &'a str) -> &'a str
// — returned &str lives as long as s

// Rule 3: If one input is &self or &mut self, output gets that lifetime
impl MyStruct {
    fn get_name(&self) -> &str { &self.name }
    // compiler sees: fn get_name<'a>(&'a self) -> &'a str
}
When these rules resolve all lifetimes, the compiler doesn't ask you to annotate anything. You only need explicit lifetime syntax when the rules are ambiguous.
✅ True or False
Most Rust functions that take and return references require explicit lifetime annotations written by the programmer.
False! Lifetime elision rules handle the common cases automatically. You typically only need to write lifetime annotations when: (1) a function takes multiple references and returns one — the compiler doesn't know which input the output borrows from; (2) a struct holds a reference; (3) you're implementing certain traits. In application code like APInox, you rarely write 'a.

3. When You DO Need Explicit Lifetimes

The compiler asks for a lifetime when it can't determine which input a returned reference borrows from:

// PROBLEM: two input references — which does the return borrow from?
fn longest(x: &str, y: &str) -> &str {
//                                 ^ ERROR: missing lifetime specifier
    if x.len() > y.len() { x } else { y }  // could be either!
}

// SOLUTION: name the lifetime, tie all three together
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
//          ^       ^               ^          ^
//     declare    both inputs    output lives as long as
//     lifetime   share 'a      the SHORTER of x or y

    if x.len() > y.len() { x } else { y }
}

// Usage:
let s1 = String::from("long string");
let result;
{
    let s2 = String::from("xyz");
    result = longest(s1.as_str(), s2.as_str());
    println!("{}", result);  // OK — both s1 and s2 alive here
}
// println!("{}", result);  // ERROR — s2 dropped, result might ref s2
🔘 Multiple Choice
In fn longest<'a>(x: &'a str, y: &'a str) -> &'a str, what does 'a actually mean at runtime?
✓ The shorter lifetime! 'a is substituted with the overlap — the shorter of the two actual lifetimes. The returned reference is only valid while both inputs are still alive. Lifetimes are purely a compile-time concept — they have zero runtime cost. They generate no code; they're just annotations that let the compiler check safety.
MAUI's memory story: everything is managed by the .NET GC, which works beautifully until you have a CollectionView binding to an ObservableCollection on a page that isn't fully disposed because the navigation stack held a reference. Memory climbs. The GC eventually collects. In the meantime your enterprise iPad app has been killed by iOS for using too much memory. 4. Lifetime Annotations in Structs

If a struct holds a reference (not an owned value), it needs a lifetime annotation:

// Struct that borrows rather than owns:
struct XmlParser<'a> {
    input: &'a str,    // borrows the input string — doesn't copy it
}

impl<'a> XmlParser<'a> {
    fn new(input: &'a str) -> Self {
        XmlParser { input }
    }

    fn parse_tag(&self) -> &'a str {
        // returned &str is a slice of self.input
        // lives as long as 'a — as long as the original string
        &self.input[1..self.input.find('>').unwrap_or(self.input.len())]
    }
}

let xml = String::from("<root>data</root>");
let parser = XmlParser::new(&xml);
let tag = parser.parse_tag();  // valid while xml exists
APInox avoids this by using owned String fields instead of &str. Most application code makes the same choice — lifetime-annotated structs are more common in high-performance library code (parsers, lexers) where you want to avoid copying large strings.
Java developers: synchronized. Also ReentrantLock. Also ReadWriteLock. Also StampedLock (Java 8, "better performance"). Also volatile. Also AtomicReference. Also ConcurrentHashMap. Choose wrong: data corruption. Choose right: you still chose wrong but it works by accident. The Stack Overflow answer for "when to use which" has 47 upvotes and ends with "it depends".
✍️ Fill in the Blank
Add the lifetime annotation to make this struct compile. The struct borrows a slice of bytes:
struct Buffer {
    data: &'a [u8],
}
<'a> — the struct declaration must name the lifetime parameter: struct Buffer<'a>. The apostrophe is mandatory — it distinguishes lifetime parameters from type parameters (<T>). By convention, single-letter names like 'a, 'b, 'r are used, or descriptive names like 'input for clarity.

5. 'static — The Special Lifetime

// 'static means: lives for the entire program duration

// String literals are always 'static — they're compiled into the binary:
let s: &'static str = "hello world";

// Required by tokio::spawn — spawned tasks can run indefinitely:
tokio::spawn(async move {
    // data used here must be 'static (owned, not borrowed)
});

// Boxed trait objects often require 'static:
fn register_handler(handler: Box<dyn Fn() + 'static>) {
    // handler stored long-term — must not hold short-lived borrows
}

// You'll see this in error messages:
// error: `data` does not live long enough, expected 'static lifetime
✅ True or False
A 'static reference always points to read-only data that can never be modified.
False! 'static means the reference is valid for the entire program — it says nothing about mutability. An owned String wrapped in Arc and stored in a lazy static satisfies 'static and is mutable (behind a Mutex). String literals (&'static str) happen to be immutable, but that's because they're in read-only memory — not because of the lifetime.
🐛 Spot the Bug
This won't compile. Click the line that causes the lifetime violation:
fn get_greeting() -> &str { let greeting = String::from("Hello!"); &greeting }
Line 3! greeting is a local String that gets dropped when the function returns. Returning &greeting would be a dangling reference — pointing to freed memory. The fix: return the owned String instead: fn get_greeting() -> String { String::from("Hello!") }. Or use a string literal: fn get_greeting() -> &'static str { "Hello!" } — string literals live forever.
Rust's Box<T>, Rc<T>, and Arc<T> give you explicit, auditable control over allocation and sharing. No dangling pointers. No use-after-free. The enterprise security team's reaction to this: require that all pointers be documented in the Security Design Document, reviewed by the Architecture Board, and approved before implementation.
