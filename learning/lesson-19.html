1. The Two Communication Channels

Tauri provides two ways for the webview and the Rust backend to communicate:

ChannelDirectionPatternUse when...
invoke() → Webview to Rust Request/response (like HTTP) Webview needs to ask Rust for something
emit() / listen() ← Rust to Webview Pub/sub events Rust needs to push data to the webview
Think of it like JavaScript's fetch() vs WebSocket. invoke() is the request-response model: the webview asks, Rust answers. Events are push-only: Rust can fire an event any time, even from a background thread, and the webview listens.

2. invoke() — Calling Rust from TypeScript

// webview/src/utils/bridge.ts — APInox's real invoke wrapper:
import { invoke } from '@tauri-apps/api/core';

// Simple call
const version = await invoke<string>('get_app_version');

// With arguments — must match the Rust function parameter names exactly
const project = await invoke<ApinoxProject>('load_project', {
    dirPath: '/Users/mark/Projects/MySOAPProject'
});

// APInox's bridge wraps invoke for consistency:
export async function invokeTauriCommand<T = any>(
    command: string,
    args?: Record<string, any>
): Promise<T> {
    await ensureTauriInitialized();
    try {
        return await tauriInvoke(command, args);
    } catch (error: any) {
        console.error(`[Bridge] Tauri command '${command}' failed:`, error);
        throw error;
    }
}
// Rust side — the Err becomes a JS rejected promise:
#[tauri::command]
pub async fn load_project(dir_path: String) -> Result<ApinoxProject, String> {
    //                                                              ^^^^^^
    // String errors are forwarded to JS as rejected promise reasons
    FolderProjectStorage::load(&PathBuf::from(dir_path))
        .map_err(|e| e.to_string())
}

In TypeScript:

try {
    const project = await invoke('load_project', { dirPath: '/bad/path' });
} catch (err) {
    // err is the String from Rust's Err(...)
    console.error('Load failed:', err);  // "No such file or directory"
}
✍️ Fill in the Blank
In TypeScript, a Tauri command that returns Err("project not found") from Rust will cause the invoke() Promise to:
The Promise will with the error string as the rejection reason.
reject — a Rust Err(String) becomes a JS rejected Promise. You catch it with try/catch around await invoke() or with .catch(). The Ok(value) path resolves the Promise with the serialized value. This maps naturally to async/await error handling in TypeScript.

3. Events — Rust Pushing to the Webview

Events are fire-and-forget messages from Rust to the webview. Any Rust code with access to an AppHandle can emit an event, even background threads:

// Rust — emit an event from any command or background task:
#[tauri::command]
pub async fn start_long_operation(
    app: tauri::AppHandle,
    params: OperationParams,
) -> Result<(), String> {
    let app_clone = app.clone();
    
    tokio::spawn(async move {
        for i in 0..10 {
            tokio::time::sleep(std::time::Duration::from_millis(500)).await;
            
            // Push progress update to ALL windows
            app_clone.emit("operation-progress", serde_json::json!({
                "step": i + 1,
                "total": 10,
                "message": format!("Processing item {}", i + 1),
            })).ok();
        }
        
        app_clone.emit("operation-complete", serde_json::json!({
            "success": true,
            "result": "All done!"
        })).ok();
    });
    
    Ok(())  // Returns immediately
}
// TypeScript — listen for events:
import { listen } from '@tauri-apps/api/event';

// Returns an unlisten function — call it to clean up!
const unlisten = await listen<{ step: number; total: number; message: string }>(
    'operation-progress',
    (event) => {
        console.log(`Step ${event.payload.step}/${event.payload.total}`);
        setProgress(event.payload.step / event.payload.total);
    }
);

await listen('operation-complete', (event) => {
    console.log('Done!', event.payload);
    unlisten(); // Stop listening to progress events
});

// Trigger the operation
await invoke('start_long_operation', { params: { ... } });
✅ True or False
app.emit("my-event", data) in Rust sends the event to a specific window only — you need to specify which window to receive it.
False! app.emit() broadcasts to ALL windows. To target a specific window use app.get_webview_window("main").unwrap().emit("my-event", data). In APInox there's only one window, so the distinction doesn't matter — but in multi-window apps (e.g., a detached editor panel) you'd want targeted events.

4. APInox's Bridge — The Dual-Mode Abstraction

APInox has an interesting historical layer: the bridge.ts file was originally designed for a sidecar (postMessage to a Node.js process). As functionality migrated to Rust, the bridge became a hybrid:

// bridge.ts — the two paths:

// PATH 1: Direct Tauri invoke (pure Rust commands)
export async function invokeTauriCommand<T>(command: string, args?: any): Promise<T> {
    await ensureTauriInitialized();
    return tauriInvoke(command, args);   // calls Rust directly
}

// PATH 2: Legacy sidecar message routing (being phased out)
export async function sendMessage(message: BridgeMessage): Promise<BackendMessage> {
    // Routes to the Node.js sidecar process via HTTP
    // ...
}

// Usage in components:
// Modern (Rust):
const project = await bridge.invokeTauriCommand('load_project', { dirPath });

// Legacy (sidecar HTTP — being migrated):
const result = await bridge.sendMessage({ command: 'parseWsdl', url });
⚠️ The dual bridge is a migration artifact. New Tauri projects should use invoke() exclusively — no sidecar needed. The pattern to learn is the invoke() path; the sendMessage() path is legacy.

5. Window Events — Lifecycle and User Actions

Beyond custom events, Tauri fires built-in window lifecycle events:

import { getCurrentWindow } from '@tauri-apps/api/window';

const appWindow = getCurrentWindow();

// Listen for window events
await appWindow.onCloseRequested(async (event) => {
    // Intercept the close — ask user to save first
    const shouldClose = await confirmUnsavedChanges();
    if (!shouldClose) {
        event.preventDefault();  // Cancel the close
    }
});

await appWindow.listen('tauri://focus', () => {
    console.log('Window focused — refresh data?');
});

await appWindow.listen('tauri://blur', () => {
    console.log('Window lost focus — pause background polling?');
});

await appWindow.listen('tauri://move', (event) => {
    console.log('Window moved to:', event.payload);
});
// In Rust, emit to specific window:
#[tauri::command]
pub fn open_project_from_args(
    app: tauri::AppHandle,
    path: String,
) {
    let window = app.get_webview_window("main").unwrap();
    // Push to webview after Rust opens/validates the path
    window.emit("open-project-path", &path).ok();
}
🔘 Multiple Choice
A Tauri app needs to warn the user before closing if there are unsaved changes. Which approach is correct?
onCloseRequested with event.preventDefault() — Tauri fires this event before the window actually closes, giving you a chance to cancel it. Note: window.onbeforeunload doesn't work in Tauri (the webview is sandboxed). You must use the Tauri API. Show your "Save changes?" dialog inside the handler and only let the close proceed if the user confirms.

🔮 Predict & Reveal
If you forget to call unlisten() after listening to a Tauri event, what happens?
// Component mounts and starts listening:
const unlisten = await listen('operation-progress', handler);

// Component unmounts — but unlisten() is never called
// What's the consequence?
Memory leak and phantom handler calls! The event listener stays alive even after the component unmounts. If the event fires again, handler is called even though the component no longer exists — potentially updating unmounted state, causing React "Can't perform a state update on unmounted component" warnings, or worse, null reference crashes. Always call unlisten() in your cleanup: in React, return it from a useEffect cleanup function: useEffect(() => { let unsub; listen(...).then(fn => unsub = fn); return () => unsub?.(); }, []);
🐛 Spot the Bug
This Rust command is supposed to emit a "done" event when a background task completes. Click the buggy line:
#[tauri::command] pub async fn start_task(app: tauri::AppHandle) -> Result<(), String> {     tokio::spawn(async {         do_long_work().await;         app.emit("task-done", true).ok();     });     Ok(()) }
Line 3! tokio::spawn(async { creates a closure that tries to capture app — but app needs to be moved into the async block to cross thread boundaries. Fix: tokio::spawn(async move {. The move keyword transfers ownership of app into the spawned task. Without it, the borrow checker rejects the code because the spawned task might outlive the function's stack frame where app lives.
