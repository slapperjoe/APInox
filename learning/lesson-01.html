1. The Entry Point

Open src-tauri/src/main.rs — it's just 4 lines:

// Prevents additional console window on Windows in release, DO NOT REMOVE!!
#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]

fn main() {
    apinox_lib::run();
}

#![cfg_attr(...)] is a crate-level attribute (the ! means "apply to the whole file"). It's a conditional compilation directive:

  • debug_assertions is true in debug builds, false in release
  • When in release mode: apply windows_subsystem = "windows" → hides the terminal window
C# equivalent: #if !DEBUG / #endif or <OutputType>WinExe</OutputType> in .csproj

2. Variables and Immutability

In Rust, variables are immutable by default. This is the opposite of C# and JS:

Stack vs Heap — Rust knows at compile time what goes where
let x = 5;         // immutable — you CANNOT reassign x
let mut y = 5;     // mutable — you CAN reassign y
y = 10;            // ✅ fine
x = 10;            // ❌ compile error!
int x = 5;        // mutable by default in C#
const int c = 5;  // immutable
✍️ Fill in the Blank
This code won't compile. Add the missing keyword to fix it:
let count = 0;    count += 1;
mut — let mut count = 0;. Without mut, Rust treats the binding as immutable and rejects the += 1 at compile time. This is intentional — it forces you to be explicit about what changes.

Shadowing — a twist you won't see in C#

You can re-declare a variable with the same name (shadowing). This is different from mutation:

let x = 5;
let x = x + 1;    // shadows previous x — creates a NEW binding
let x = x * 2;    // shadows again

println!("{}", x); // 12

Useful when you want to transform a value and reuse the name — e.g., parsing a string into a number without a different variable name.

3. Types

Strings — the most confusing part for newcomers

TypeDescriptionC# analogy
StringOwned, heap-allocated, growablestring (System.String)
&strBorrowed slice — a view into string dataReadOnlySpan<char>
 
let owned: String = String::from("hello");   // heap-allocated, you own it
let borrowed: &str = "hello";                // string literal — lives in binary
let also_borrowed: &str = &owned;           // view into the owned string
Rule of thumb: Functions that receive strings usually take &str (flexible, no ownership transfer). Functions that create or return strings usually return String (caller can own it).
✅ True or False
&str and String are the same type in Rust.
False! String is an owned, growable heap string. &str is a borrowed view — it doesn't own the data, and you can't grow it. You can borrow a &str from a String with &my_string, but they are distinct types.

4. Ownership — Rust's Big Idea

Rust has no garbage collector and no manual free(). Instead, every value has exactly one owner. When the owner goes out of scope, the value is automatically freed.

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;    // s1 is MOVED into s2 — s1 is no longer valid!

    println!("{}", s1); // ❌ compile error: value borrowed after move
    println!("{}", s2); // ✅ fine
}
An architect will hear "ownership model" and immediately schedule a 2-hour workshop to rename it the "Resource Lifecycle Governance Framework". The resulting diagram will have 14 boxes. Three will say "TBD". One will say "Event Bus?" with a question mark. The outcome will be a Confluence page nobody reads and a follow-up meeting to "align on next steps".
C# comparison: In C#, s2 = s1 gives you two references to the same heap object. In Rust, the value moves — only one variable owns it at a time.
🐛 Spot the Bug
Click the line that causes a compile error:
let name = String::from("Alice"); let greeting = name; println!("Hello, {}", name);
Line 3! name was moved into greeting on line 2. After the move, name is invalid — you can't use it. Fixes: let greeting = name.clone() (deep copy) or let greeting = &name (borrow instead).

Borrowing with &

To use a value in a function without giving up ownership, you borrow it:

fn print_length(s: &String) {   // borrows — doesn't take ownership
    println!("Length: {}", s.len());
}   // s drops here, but we didn't own it — nothing freed

fn main() {
    let owned = String::from("hello");
    print_length(&owned);   // borrow with &
    println!("{}", owned);  // ✅ still valid!
}

What does the & symbol mean in a function parameter like fn foo(s: &String)?
✓ Borrow! &T is an immutable reference. The function can read the value but not modify or free it. The original owner retains ownership — the value is still valid after the call.
Rust eliminates an entire class of memory vulnerabilities at compile time — buffer overflows, use-after-free, null pointer dereferences. This is the stuff that fills CVE databases. 5. The Module System

In lib.rs you'll see module declarations at the top:

mod project_storage;          // loads src/project_storage.rs
mod history_storage;
pub mod settings_manager;     // public module
pub mod utils;                // loads src/utils/mod.rs
pub mod soap;                 // loads src/soap/mod.rs
pub mod parsers;
DeclarationFile it loads
mod project_storage;src/project_storage.rs
pub mod soap;src/soap/mod.rs (folder module)

To use items from another module:

use crate::soap::SoapClient;              // absolute — from crate root
use super::ws_security::WsSecurityConfig; // relative — up one level
Oracle Siebel manages object lifecycles by writing intermediate state to 47 database tables on every user interaction. Memory is someone else's problem when you have Oracle Exadata. The Siebel "ownership model" is: BC owns the BO, the BO owns the Applet, the Applet owns your evenings. You do not own any of this. Siebel owns you.
🤔 Predict the Output
What happens when you try to compile this code? Think before revealing.
fn main() {
    let s = String::from("hello");
    let r1 = &s;
    let r2 = &s;
    println!("{} and {}", r1, r2);
}
✅ It compiles and prints: hello and hello

You can have multiple immutable references (&) at the same time — that's fine! The borrow rule only prevents having a mutable reference (&mut) at the same time as any other reference.

6. Attributes at a Glance

Attributes use #[...] syntax. You'll see them on almost everything:

#[tauri::command]               // marks a function as a Tauri IPC command
pub async fn quit_app(...) { }

#[cfg(target_os = "macos")]     // only compile the next item on macOS
use tauri_plugin_decorum::...;

#[cfg(windows)]                 // only on Windows
use windows::Win32::...;

#[derive(Debug, Clone)]         // auto-generate Debug and Clone implementations
pub struct Foo { }

#[cfg(...)] is Rust's compile-time conditional — the excluded code is completely absent from the binary. Zero overhead.
