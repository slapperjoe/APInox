1. The Build Pipeline

A Tauri production build compiles everything into a single native installer. The pipeline has four stages:

1. Frontend build      → npm/vite builds React → webview/dist/
2. Sidecar compile     → tsc + pkg → apinox-sidecar binary (if used)
3. Rust compile        → cargo build --release → libapinox.so / .dylib / .dll
4. Tauri bundle        → packs webview + Rust into installer / app bundle
# The single command that does all of this:
npm run tauri:build
# or
npx tauri build

# Output (from tauri.conf.json bundle.targets):
# macOS:   src-tauri/target/release/bundle/dmg/APInox_0.16.114_x64.dmg
# Windows: src-tauri/target/release/bundle/nsis/APInox_0.16.114_x64-setup.exe
# Linux:   src-tauri/target/release/bundle/deb/apinox_0.16.114_amd64.deb

APInox also has a beforeBuildCommand in tauri.conf.json that runs the webview build automatically before Tauri bundles:

{
  "build": {
    "frontendDist": "webview/dist",
    "beforeBuildCommand": "cd src-tauri/webview && npm run build:skip-check"
  }
}

3. Code Signing

Without code signing, OS security features warn users or block installation. Here's what each platform requires:

PlatformRequirementCost
macOSApple Developer certificate + notarization$99/year
WindowsEV code signing certificate$300-500/year
LinuxNo signing required (GPG for packages optionally)Free
 
# macOS signing via environment variables (set in CI):
export APPLE_CERTIFICATE="base64-encoded-p12-cert"
export APPLE_CERTIFICATE_PASSWORD="cert-password"
export APPLE_ID="developer@example.com"
export APPLE_PASSWORD="app-specific-password"
export APPLE_TEAM_ID="XXXXXXXXXX"

# Windows signing:
export WINDOWS_CERTIFICATE="base64-encoded-pfx"
export WINDOWS_CERTIFICATE_PASSWORD="cert-password"

# Then just:
npx tauri build
# Tauri automatically signs and notarizes using the env vars
✅ True or False
An unsigned macOS Tauri app can be installed by users by holding Option and double-clicking, but macOS Gatekeeper will block it by default.
True! Gatekeeper blocks unsigned apps with "cannot be opened because the developer cannot be verified." Users can override this in System Settings → Privacy & Security → "Open Anyway" (Control+click also works). For internal tools distributed to known users, this is an acceptable tradeoff. For public distribution, signing is essential — most users won't know how to override Gatekeeper.

4. Cross-Platform CI with GitHub Actions

The standard approach for Tauri apps: run the build on each platform's native runner:

# .github/workflows/release.yml
name: Release
on:
  push:
    tags: ['v*']        # triggers on git tags like v1.2.3

jobs:
  build:
    strategy:
      matrix:
        platform:
          - os: ubuntu-latest
            rust-target: x86_64-unknown-linux-gnu
          - os: windows-latest
            rust-target: x86_64-pc-windows-msvc
          - os: macos-latest
            rust-target: x86_64-apple-darwin
          - os: macos-latest         # Apple Silicon
            rust-target: aarch64-apple-darwin

    runs-on: ${{ matrix.platform.os }}
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with: { node-version: 20 }

      - name: Setup Rust
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.platform.rust-target }}

      - name: Install dependencies
        run: npm install

      - name: Build Tauri app
        uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
          APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
          APPLE_ID: ${{ secrets.APPLE_ID }}
          APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
          APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
        with:
          tagName: ${{ github.ref_name }}
          releaseName: 'APInox ${{ github.ref_name }}'
          args: --target ${{ matrix.platform.rust-target }}
🔘 Multiple Choice
Why does the CI matrix include two macOS entries (x86_64 and aarch64)?
Two CPU architectures! Apple Silicon (M1/M2/M3) uses ARM (aarch64), while older Macs use Intel x86_64. A binary compiled for one won't run on the other natively (though Rosetta 2 can translate Intel binaries on Apple Silicon with a performance hit). For maximum performance on both, you build two separate binaries. Some teams build a universal binary with lipo — it contains both architectures and runs natively on either.
Oracle WebLogic deployment: package your app as an EAR file (ZIP of WARs of JARs), upload via the WebLogic console, undeploy the old version (4 minutes), deploy the new one (4 minutes), restart if you changed a shared library (12 minutes), discover you needed to restart anyway (restart: 12 minutes). A config change requires a redeploy. A property file change requires a redeploy. Breathing near the server may require a redeploy. This is 'enterprise application server architecture'.

5. Auto-Updater

Tauri v2 includes a built-in updater plugin for delivering updates to users:

# Cargo.toml
[dependencies]
tauri-plugin-updater = "2"
// In setup:
.plugin(tauri_plugin_updater::Builder::default().build())
// TypeScript — check for and apply updates:
import { check } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';

async function checkForUpdates() {
    const update = await check();
    if (update) {
        console.log(`Update available: ${update.version}`);
        console.log(`Release notes: ${update.body}`);

        // Download and install
        await update.downloadAndInstall((event) => {
            if (event.event === 'Progress') {
                setProgress(event.data.chunkLength / event.data.contentLength);
            }
        });

        // Restart to apply
        await relaunch();
    }
}
// tauri.conf.json — point to your update server:
{
  "plugins": {
    "updater": {
      "endpoints": [
        "https://releases.myapp.com/{{target}}/{{arch}}/{{current_version}}"
      ],
      "dialog": true,      // show native update dialog
      "pubkey": "your-public-key-for-update-verification"
    }
  }
}
The updater signs update bundles cryptographically — users can't be tricked into installing a malicious "update." You sign with a private key (kept secret); the public key is embedded in the app. APInox doesn't currently use the built-in updater (it's distributed manually), but this is the standard pattern for apps with auto-update.

6. Resources & Sidecar Bundling

APInox bundles a Node.js sidecar binary alongside the Rust app. This is configured in tauri.conf.json:

{
  "bundle": {
    "resources": {
      "../CHANGELOG.md": "CHANGELOG.md"
    },
    "externalBin": [
      "binaries/apinox-sidecar"   // sidecar binary (built separately)
    ]
  }
}

The sidecar binary must exist for all target platforms. For APInox:

# Build the Node.js sidecar into a standalone binary:
npm run build:sidecar     # tsc compilation
npm run prepare:sidecar   # pkg bundles into apinox-sidecar binary

# The naming convention Tauri expects (platform suffix):
# binaries/apinox-sidecar-x86_64-pc-windows-msvc.exe   (Windows x64)
# binaries/apinox-sidecar-x86_64-apple-darwin           (macOS Intel)
# binaries/apinox-sidecar-aarch64-apple-darwin          (macOS M1/M2)
🐛 Spot the Bug
This CI workflow is meant to build for macOS Apple Silicon. Click the buggy line:
- name: Setup Rust   uses: dtolnay/rust-toolchain@stable   with:     targets: aarch64-apple-darwin - name: Build   run: npx tauri build   # Missing the --target flag!
Line 4 is misleading but the real issue is line 6! Installing the aarch64-apple-darwin toolchain is correct — but without passing --target aarch64-apple-darwin to tauri build, it compiles for the runner's native architecture (x86_64 on a standard GitHub runner). Fix: npx tauri build --target aarch64-apple-darwin. The toolchain install just makes the target available; you must explicitly request it.
