String Types

⏱ ~25 min 📁 project_storage.rs, secret_storage.rs, settings_manager.rs

0 of 5 questions answered

Strings are where most C# developers hit their first Rust wall. The reason there are two string types is ownership — once you understand that, it all clicks.

1. String vs &str — The Core Split

String&str
OwnershipOwned — heap allocatedBorrowed — just a view
Mutable?Yes (if declared mut)No
C# analogyStringBuilder + stringReadOnlySpan<char>
Size3 words on stack (ptr, len, capacity)2 words on stack (ptr, len)
Create from literal"hello".to_string()"hello" (already a &str)
String owns heap memory; &str is a view into existing data (stack literal, String, or anywhere)
let owned: String = String::from("hello");     // owns heap memory
let owned: String = "hello".to_string();        // same thing
let owned: String = "hello".to_owned();         // also same

let borrowed: &str = "hello";                  // pointer into read-only memory
let borrowed: &str = &owned;                   // borrow from a String
let borrowed: &str = &owned[0..3];             // slice — "hel"

The golden rule: prefer &str in function parameters

// ✅ Good — accepts both String and &str
fn greet(name: &str) {
    println!("Hello, {}!", name);
}

// ❌ Restrictive — only accepts owned String
fn greet(name: String) {
    println!("Hello, {}!", name);
}

// Calling the good version with either type:
greet("literal");                  // &str literal — works
greet(&my_string);                // &String auto-coerces to &str — works
greet(my_string.as_str());         // explicit — also works
From project_storage.rs: Functions take &str or String at the Tauri command boundary where JSON deserialization gives you owned Strings. Inside the function, &str slices are used freely.

🔘 Multiple Choice
Why should function parameters usually take &str rather than String?
✓ Flexibility! A &str parameter accepts string literals directly ("hello"), borrows of String via auto-deref (&my_string), and string slices. A String parameter forces callers to either already own a String or call .to_string() just to call your function — unnecessary allocation.

2. String Conversions — The Most Common Operations

Java has String (immutable), StringBuilder (mutable, single-thread), StringBuffer (mutable, thread-safe, slower), StringJoiner (Java 8, because apparently String.join() wasn't enough), and CharSequence (interface for all of the above). Rust has String and &str. You'll get it in a day.
let s: String = "hello".to_string();
let s: String = "hello".to_owned();
let s: String = String::from("hello");
let s: String = format!("hello {}", name);      // like String.Format or $""

// String → &str
let slice: &str = &s;
let slice: &str = s.as_str();

// Number → String
let n = 42;
let s: String = n.to_string();
let s: String = format!("{}", n);

// String → Number
let n: i32 = "42".parse().unwrap();
let n: i32 = "42".parse::<i32>().unwrap();     // turbofish syntax

// Concatenation
let combined = format!("{} {}", first, last);   // ✅ preferred
let combined = first + " " + &last;             // ✅ moves first, borrows last
// combined = first + last;                     // ❌ type error — + needs &str rhs
✍️ Fill in the Blank
Convert the number port to a String for use in a URL:
let url = format!("http://localhost:{}", port);
Nothing needed! format! accepts any type that implements Display — numbers, booleans, floats — directly. You don't need to call .to_string() first. format!("http://localhost:{}", port) is the idiomatic way. If you answered .to_string(), that also works but is redundant here.

3. PathBuf and Path — Strings for File Paths

File paths are their own type pair in Rust — analogous to String/&str:

PathBufPath (&Path)
RelationLike String — ownedLike &str — borrowed view
C# analogynew FileInfo("path")string path

From project_storage.rs:

use std::path::PathBuf;

// Build paths cross-platform (handles / and \ automatically):
let config_dir = PathBuf::from(&dir_path);
let props_path = config_dir.join("properties.json");   // appends segment
let sub_dir    = config_dir.join("interfaces").join("MyService");

// Check existence:
if props_path.exists() { ... }
if props_path.is_file() { ... }
if config_dir.is_dir() { ... }

// Get parts:
let name = path.file_name().unwrap().to_string_lossy();
let ext  = path.extension().unwrap().to_str().unwrap();
let dir  = path.parent().unwrap();

// Convert back to string when needed (e.g., for logging):
let s: &str = path.to_str().unwrap();              // fails if non-UTF8
let s: String = path.to_string_lossy().into_owned(); // always works
✅ True or False
You should use String and manual "/" concatenation to build file paths in cross-platform Rust code.
False! Always use PathBuf::join(). Windows uses \, Unix uses /. join() uses the correct separator automatically. format!("{}/{}", dir, file) will break on Windows. PathBuf::from(dir).join(file) works everywhere — which matters for a cross-platform Tauri app like APInox.

4. Common String Methods

let s = String::from("  Hello, World!  ");

// Trimming
s.trim()                    // "Hello, World!" — both ends
s.trim_start()              // "Hello, World!  "
s.trim_end()                // "  Hello, World!"

// Searching
s.contains("World")         // true
s.starts_with("Hello")      // false — whitespace!
s.find("World")             // Some(9) — byte index

// Splitting
let parts: Vec<&str> = "a,b,c".split(',').collect();
let parts: Vec<&str> = "a,b,c".splitn(2, ',').collect(); // ["a", "b,c"]

// Replacing
s.replace("World", "Rust")
s.replacen("l", "L", 1)    // replace first occurrence only

// Case
s.to_lowercase()
s.to_uppercase()

// Lines
let lines: Vec<&str> = text.lines().collect();
🐛 Spot the Bug
This code tries to get the 3rd character of a string. Click the buggy line:
let s = String::from("hello"); let ch = s[2]; println!("{}", ch);
Line 2! Rust does not allow indexing into a String with []. This is intentional — a String is UTF-8, and s[2] would be a byte index, not a character index. For multi-byte characters (e.g., emoji, accented letters), this would give you a partial character. Instead:

s.chars().nth(2) → Option<char> — correct, counts characters
&s[0..3] → &str slice — byte range, panics on non-char boundary
s.as_bytes()[2] → u8 — explicit byte access

5. The format! Macro — Your Swiss Army Knife

// Basic — like C# $"" strings
let msg = format!("Port: {}", port);
let msg = format!("Host: {host}");          // Rust 1.58+ — variable name directly

// Debug output
let msg = format!("{:?}", my_struct);       // requires #[derive(Debug)]
let msg = format!("{:#?}", my_struct);      // pretty-printed (multiline)

// Number formatting
format!("{:05}", 42)       // "00042" — zero-padded to width 5
format!("{:.2}", 3.14159)  // "3.14" — 2 decimal places
format!("{:>10}", "hi")    // "        hi" — right-aligned in 10 chars
format!("{:x}", 255)       // "ff" — hex
format!("{:b}", 7)         // "111" — binary

// From secret_storage.rs — building error messages:
return Err(format!("Could not determine config directory for platform: {}", std::env::consts::OS));
🤔 Predict the Output
What does this print?
let name = "APInox";
let version = 2;
println!("Welcome to {} v{:03}!", name, version);
Welcome to APInox v002!
{:03} formats the integer with a minimum width of 3, zero-padded. So 2 becomes 002. Format specifiers go after : inside the braces: {value:specifier}.
