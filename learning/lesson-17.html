What Is Tauri?

Tauri is a framework for building desktop apps with a web frontend (React, Vue, etc.) and a Rust backend. Think of it as Electron, but written in Rust — the result is smaller binaries (often 10-20MB vs 100MB+), better security, and native performance for backend work.

Rather than bundling a full Chromium runtime like Electron does, Tauri uses the OS's native webview:

PlatformWebview engine
macOSWKWebView (same as Safari)
WindowsWebView2 (Chromium-based, from Edge)
LinuxWebKitGTK
⚡ CSS consistency caveat: Because each platform uses a different engine, edge cases in CSS can look different. Test on your target platforms — especially scroll behavior, animations, and newer CSS features.

The Three-Layer Architecture

┌─────────────────────────────────────────────────────┐
│                  Tauri Desktop App                   │
├─────────────────────────────────────────────────────┤
│  🌐 Webview Layer (React / HTML / CSS / TypeScript) │
│  • Renders your UI — isolated in an OS sandbox      │
│  • Calls Rust via invoke() — can't touch OS directly│
├─────────────────────────────────────────────────────┤
│  🦀 Tauri Core Layer (Rust)                         │
│  • Event loop, window management                    │
│  • Exposes #[tauri::command] functions to webview   │
│  • Manages plugins, file system, native APIs        │
├─────────────────────────────────────────────────────┤
│  🔌 OS / Native APIs                               │
│  • File system, clipboard, notifications            │
│  • System tray, dialogs, shell commands             │
└─────────────────────────────────────────────────────┘

In APInox's case there's also a legacy sidecar (a Node.js process), though the codebase is actively migrating that logic into the Rust layer:

// src-tauri/src/lib.rs — sidecar process management
static SIDECAR_PORT: AtomicU16 = AtomicU16::new(0);
static SIDECAR_PROCESS: Mutex<Option<Child>> = Mutex::new(None);

fn spawn_sidecar(app_handle: &tauri::AppHandle) -> Result<(), String> {
    // Launches Node.js as a child process — a separate OS process
    let mut cmd = Command::new("node");
    cmd.arg("sidecar/dist/index.js")
       .stdout(Stdio::piped());
    // ...
}
The sidecar pattern is unusual — most Tauri apps don't need it. APInox used it to port existing Node.js SOAP logic before rewriting in Rust. New Tauri apps: put logic in Rust directly.
🔘 Multiple Choice
What does Tauri use to render the UI on Windows, instead of bundling Chromium?
WebView2 — on Windows, Tauri uses the WebView2 runtime that ships with Microsoft Edge and Windows 11. This is why Tauri installers are so small — the rendering engine is already on the machine. On macOS it's WKWebView (Safari's engine), on Linux it's WebKitGTK. The downside is slight CSS behavior differences across platforms.

The Entry Point — run()

Tauri apps start in main.rs which immediately calls into lib.rs:

// src-tauri/src/main.rs
#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]
// ↑ On Windows release builds, don't spawn a console window

fn main() {
    apinox_lib::run();
}

// src-tauri/src/lib.rs — the actual setup:
pub fn run() {
    tauri::Builder::default()
        // Register plugins (file dialog, clipboard, etc.)
        .plugin(tauri_plugin_dialog::init())
        .plugin(tauri_plugin_clipboard_manager::init())
        .plugin(tauri_plugin_shell::init())
        // Register all Rust functions callable from the webview
        .invoke_handler(tauri::generate_handler![
            quit_app,
            get_app_version,
            project_storage::save_project,
            soap::commands::execute_soap_request,
            testing::commands::run_test_case,
            // ... 40+ more commands
        ])
        // Setup callback — runs before the window appears
        .setup(|app| {
            // Initialize logging, spawn sidecar, etc.
            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}

The generate_handler! macro generates a dispatch table. When the webview calls invoke('save_project', { ... }), Tauri looks up project_storage::save_project in that table and calls it with the provided JSON arguments deserialized into Rust types.

generate_handler! — this macro takes a list of function names (all annotated with #[tauri::command]) and produces the runtime dispatch table. If you forget to add a command here, calling it from the webview returns an error even though the Rust function exists. This is how Tauri creates the "allowed commands" surface area.

Window Configuration — tauri.conf.json

The window and app configuration lives in src-tauri/tauri.conf.json. Here's APInox's actual config:

{
  "productName": "APInox",
  "version": "0.16.114",
  "identifier": "app.apinox.dev1",
  "build": {
    "frontendDist": "webview/dist",
    "devUrl": "http://localhost:5173",
    "beforeDevCommand": "cd src-tauri/webview && npm run dev",
    "beforeBuildCommand": "cd src-tauri/webview && npm run build:skip-check"
  },
  "app": {
    "windows": [
      {
        "title": "APInox",
        "width": 1400, "height": 900,
        "minWidth": 800, "minHeight": 600,
        "titleBarStyle": "Overlay",   // ← macOS traffic lights in content
        "decorations": true
      }
    ],
    "security": {
      "csp": "default-src 'self'; connect-src 'self' https: http://localhost:*;"
    }
  },
  "bundle": {
    "targets": ["nsis", "dmg"],       // Windows NSIS installer + macOS disk image
    "icon": ["icons/32x32.png", "icons/icon.icns", "icons/icon.ico"]
  }
}

The beforeDevCommand and beforeBuildCommand are shell commands run by Tauri before starting — used to compile the React frontend with Vite.

✅ True or False
A Tauri app's window configuration (size, title bar style, etc.) can be changed by editing tauri.conf.json without touching Rust code.
True! Most window settings are data-driven via tauri.conf.json. You can also configure multiple windows, specify which URLs are allowed, set the minimum dimensions, and toggle decorations — all without touching Rust. Some dynamic things (like programmatically creating windows at runtime) do require Rust code, but static configuration is purely JSON.

The Setup Callback — Running Before the Window Shows

.setup(|app| {
    // ← 'app' is &mut tauri::App — you get one shot at this

    // Register managed state (shared data accessible in commands)
    app.manage(MyState { counter: Mutex::new(0) });

    // Get the app's data directory
    let data_dir = app.path().app_data_dir()?;

    // Show the window only after we're ready
    let window = app.get_webview_window("main").unwrap();
    window.show()?;

    // Platform-specific tweaks
    #[cfg(target_os = "macos")]
    window.set_traffic_lights_position(14.0, 22.0);

    Ok(())
})

The setup closure runs synchronously before the event loop starts. If it returns an Err, the app exits. This is where you:

  • Register shared state with app.manage()
  • Set up logging
  • Start background services
  • Apply platform-specific customisations

Platform-Specific Code

Tauri apps frequently need to behave differently per OS. Rust's #[cfg(target_os = "...")] makes this clean:

// lib.rs — real APInox patterns:
#[cfg(target_os = "macos")]
use tauri_plugin_decorum::WebviewWindowExt;  // macOS-only traffic lights

#[cfg(windows)]
use std::os::windows::process::CommandExt;  // Windows-only process flags

#[cfg(windows)]
use windows::Win32::Graphics::Dwm::{DwmSetWindowAttribute, DWMWA_BORDER_COLOR};

// Inside setup:
#[cfg(target_os = "macos")]
{
    window.make_vibrancy(...);  // macOS frosted glass effect
}

#[cfg(windows)]
{
    // Set custom window border color via DWM
    set_window_border(&window, "#1a6ebd");
}

// In Cargo.toml — OS-specific dependencies:
// [target.'cfg(windows)'.dependencies]
// windows = { version = "0.58", features = ["Win32_Graphics_Dwm"] }
🔮 Predict & Reveal
Given this code, what happens when you build this app for macOS?
#[cfg(windows)]
fn apply_custom_chrome(window: &Window) {
    set_window_border(window, "#1a6ebd");
    set_rounded_corners(window);
}

#[cfg(not(windows))]
fn apply_custom_chrome(_window: &Window) {
    // no-op
}

// In setup:
apply_custom_chrome(&main_window);

Think about it — then reveal.

The #[cfg(not(windows))] version compiles — the no-op function. apply_custom_chrome is called, does nothing, and that's fine. The Windows-specific DWM code is entirely absent from the macOS binary — it doesn't exist, it doesn't need to link, and macOS users never see it. This is conditional compilation at the function level, not runtime if/else. Both functions have the same signature so the call site is identical on all platforms.
