⏱ ~30 min 📁 workflow/engine.rs, testing/test_runner.rs

1. tokio::spawn — Fire and Forget

Spawning a task is like C#'s Task.Run() — the task runs concurrently on the Tokio thread pool:

use tokio::task::JoinHandle;

// Spawn returns a JoinHandle — you can await it later:
let handle: JoinHandle<String> = tokio::spawn(async {
    // runs concurrently
    expensive_operation().await
});

// Do other work concurrently...
other_work().await;

// Then wait for the spawned task:
let result = handle.await?;  // propagates panic as JoinError

// Spawn and forget (fire-and-forget):
tokio::spawn(async {
    send_telemetry().await;
    // result ignored — task runs until done
});

// From workflow/engine.rs — running steps in parallel:
use tokio::task::JoinSet;

let mut set = JoinSet::new();
for step in parallel_steps {
    set.spawn(async move { execute_step(step).await });
}
// Collect results as they complete:
while let Some(result) = set.join_next().await {
    results.push(result??);
}
✍️ Fill in the Blank
Complete the spawn call. The task must own step because it may outlive the current scope:
set.spawn(async {
    execute_step(step).await
});
move — the move keyword transfers ownership of step into the async block. Without it, the closure borrows step, but the spawned task can outlive the current stack frame where step lives. Tokio requires spawned tasks to be 'static — meaning they can't borrow short-lived values. move is the fix.

2. join! — Run Tasks Concurrently, Wait for All

// Sequential — each awaits before starting the next:
let users = fetch_users().await?;
let orders = fetch_orders().await?;
let config = fetch_config().await?;
// Total time: time(users) + time(orders) + time(config)

// Concurrent with join! — all start simultaneously:
let (users, orders, config) = tokio::join!(
    fetch_users(),
    fetch_orders(),
    fetch_config(),
);
// Total time: max(time(users), time(orders), time(config))

// join! with error handling — try_join! stops on first error:
let (users, orders) = tokio::try_join!(
    fetch_users(),
    fetch_orders(),
)?;
// If either fails, the whole thing returns Err
// C# equivalent:
var usersTask = FetchUsersAsync();
var ordersTask = FetchOrdersAsync();
await Task.WhenAll(usersTask, ordersTask);
var users = await usersTask;
var orders = await ordersTask;
🤔 Predict the Output
Which is faster — sequential awaits or tokio::join! — and roughly how much faster for three 1-second operations?
join! is ~3× faster! Sequential: 1s + 1s + 1s = 3s. join!: all three start simultaneously, total ≈ 1s (the longest). This is the core win of async programming — overlapping I/O wait times. Note: this only applies to I/O-bound tasks (network, disk). CPU-bound tasks still take the same CPU time — you'd need multiple threads (tokio::spawn on a multi-threaded runtime) for true parallel CPU work.

3. Channels — Communicating Between Tasks

use tokio::sync::mpsc;  // multi-producer, single-consumer

// Create a channel with buffer size 32:
let (tx, mut rx) = mpsc::channel::<String>(32);

// Producer task — can clone tx for multiple senders:
let tx2 = tx.clone();
tokio::spawn(async move {
    tx2.send("result from task 1".to_string()).await.unwrap();
});

tokio::spawn(async move {
    tx.send("result from task 2".to_string()).await.unwrap();
});

// Consumer — receive until channel closes (all tx clones dropped):
while let Some(msg) = rx.recv().await {
    println!("Got: {}", msg);
}
Channel typeSendersReceiversUse case
mpscMany (clone tx)OneWorkers → aggregator (most common)
oneshotOneOneSingle response to a request
broadcastOneManyEvents to multiple subscribers
watchOneManyLatest-value notification (config updates)
🔘 Multiple Choice
What does "mpsc" stand for?
✓ Multi-producer, single-consumer! Many tasks can clone the tx sender and send messages. Only one task owns the rx receiver. This maps to a classic pattern: a pool of worker tasks producing results, funnelled into one aggregator task that collects them. APInox's workflow engine uses this to collect results from parallel step execution.

4. select! — Race Multiple Futures

use tokio::select;
use tokio::time::{sleep, Duration};

// Race: whichever future completes first wins
select! {
    result = fetch_data() => {
        println!("Got data: {:?}", result);
    }
    _ = sleep(Duration::from_secs(5)) => {
        println!("Timeout! fetch took too long");
    }
}

// Cancellation pattern — stop when a shutdown signal arrives:
let (shutdown_tx, mut shutdown_rx) = tokio::sync::oneshot::channel::<()>();

loop {
    select! {
        work = queue.recv() => {
            if let Some(item) = work { process(item).await; }
        }
        _ = &mut shutdown_rx => {
            println!("Shutting down");
            break;
        }
    }
}
✅ True or False
When one branch of select! completes, Tokio waits for the other branches to finish before continuing.
False! select! cancels the other branches as soon as one completes. That's the whole point — it's a race. The losing futures are dropped (cancelled). This is how timeouts work: select! between your operation and a timer — if the timer wins, the operation future is dropped and the operation stops. Be careful with non-cancellation-safe operations.

5. The Send and Sync Traits

You'll see these in compiler errors. They're auto-implemented marker traits:

TraitMeansImplemented by
SendSafe to transfer to another threadMost types (not Rc, raw pointers)
SyncSafe to reference from multiple threadsMost types (not RefCell, Cell)
 
// tokio::spawn requires the future to be Send:
tokio::spawn(async move {
    // Everything used here must be Send
    // Rc is NOT Send — use Arc instead
    // MutexGuard held across .await is problematic
});

// If you hold a std::sync::MutexGuard across .await:
let guard = mutex.lock().unwrap();
do_something_with(&guard);
some_async_call().await;  // ERROR: MutexGuard is not Send
drop(guard);              // FIX: drop before awaiting

// Use tokio::sync::Mutex instead — it's designed for async:
let guard = tokio_mutex.lock().await;
some_async_call().await;  // OK — tokio's guard is Send
🔗 Match It
Match the Tokio tool to its purpose:
All matched!
• join! → concurrent, wait for all (like Task.WhenAll)
• select! → race, first wins, rest cancelled (like Task.WhenAny + cancellation)
• mpsc::channel → producer/consumer message passing across tasks
• JoinSet → dynamic collection of spawned tasks, results collected as they arrive
