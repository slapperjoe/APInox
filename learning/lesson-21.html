Capabilities & Security

⏱ ~25 min 📁 capabilities/default.json, tauri.conf.json

0 of 5 questions answered

1. Tauri v2's Security Philosophy

Tauri's #1 design goal is security by default. The webview is a browser — and browsers have had decades of vulnerabilities. Tauri's answer is the capabilities system: every native API call must be explicitly allowed before the webview can use it.

This is different from Electron, where the webview has Node.js access by default. In Tauri:

  • ❌ The webview cannot read files, run shell commands, or access the clipboard unless you declare it
  • ✅ You declare what each window is allowed to do in capabilities/*.json
  • ✅ Permissions can be scoped (e.g., only allow reading from $HOME/Documents/**)
⚡ Why this matters: If a malicious script is injected into the webview (via XSS), it can only do what the capabilities allow. In an app that never needs file system access, the attacker's injected script literally cannot read your files — the capability doesn't exist.

2. Capabilities Files — APInox's Real Config

// src-tauri/capabilities/default.json — APInox's actual permissions:
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "enables the default permissions",
  "windows": ["main"],          // applies to the "main" window
  "permissions": [
    "core:default",             // basic window operations (title, size, etc.)
    "core:window:allow-close",
    "core:window:allow-minimize",
    "core:window:allow-maximize",
    "core:window:allow-set-focus",
    "core:window:allow-start-dragging",
    "core:window:allow-toggle-maximize",
    "decorum:allow-show-snap-overlay",  // 3rd party plugin permission
    "dialog:default",           // standard dialog operations
    "dialog:allow-save",        // specifically allow save dialogs
    "opener:default",
    {
      "identifier": "opener:allow-open-path",
      "allow": [
        { "path": "$TEMP/**" }  // scoped: can only open files in temp dir
      ]
    }
  ]
}

The windows array says which windows this capability applies to. With multiple windows, you can have different permission sets per window — your main editor window might have file access; a sandboxed preview window might have nothing.

🔘 Multiple Choice
In Tauri v2, what happens if the webview calls a plugin API (like open() from plugin-dialog) but the permission hasn't been declared in the capabilities file?
Runtime error! Tauri enforces permissions at the IPC layer. If the permission isn't declared, the command returns an error that propagates to your TypeScript as a rejected Promise. You'll see something like: "Permission denied: dialog:open-file is not allowed". This is intentional — the failure is loud so you notice it during development, rather than silently leaking access.
Upon seeing the 15-line capabilities file, an architect will ask if permissions should be managed by a centralised Policy Engine, fed by a Rules Service, with an Audit Trail Service, behind an API Gateway, governed by a Security Review Board that meets bi-weekly. The capabilities file takes 10 minutes to write. The Policy Engine Architecture takes 6 months to spec, never gets built, and the team goes back to the capabilities file.

3. CSP — Content Security Policy

The Content Security Policy in tauri.conf.json controls what the webview HTML/JS can do — what scripts can run, where network requests can go, etc. APInox's CSP:

{
  "app": {
    "security": {
      "csp": "default-src 'self'; connect-src 'self' https: http://localhost:* http://127.0.0.1:*; style-src 'self' 'unsafe-inline'; script-src 'self' 'unsafe-eval'; worker-src blob:; font-src 'self' data:; img-src 'self' data:;"
    }
  }
}

Breaking down APInox's CSP:

DirectiveValueMeaning
default-src'self'Default: only load from app bundle
connect-srchttps: http://localhost:*Can make HTTP requests to any HTTPS URL + localhost (for SOAP calls!)
style-src'unsafe-inline'Inline style attributes allowed (styled-components needs this)
script-src'unsafe-eval'Dynamic eval allowed (some bundlers need this)
img-src'self' data:Images from bundle + data URIs (base64)
⚠️ 'unsafe-inline' and 'unsafe-eval' weaken CSP. APInox needs them for styled-components and the bundler. For maximum security, avoid these — use hashes instead of inline styles and avoid eval. For an internal tool like APInox, the tradeoff is acceptable.
✍️ Fill in the Blank
APInox can make SOAP requests to https://api.company.com/soap from the webview because which CSP directive allows it?
The -src directive includes https:, which permits outbound HTTP connections to any HTTPS URL.
connect-src — this CSP directive controls where fetch(), XMLHttpRequest, and WebSocket connections can go. https: is a broad wildcard allowing all HTTPS origins. Without this, every external SOAP endpoint would be blocked by the browser's CSP enforcement. This is also why http://localhost:* is included — APInox's built-in proxy and mock servers run on local ports and the webview needs to reach them.
Enterprise security theater: your app passes the pen test. The pen test report has 47 findings. Finding #1 is 'TLS 1.0 enabled' — been in the report for 3 years. Finding #3 is 'password complexity below policy' — the policy was written in 2009 when the minimum was 6 characters. The dev team fixed it 18 months ago. The pen testers are using last year's report. Finding #12 is 'X-Frame-Options not set' — for an internal desktop tool with no browser access. The CIO presents the report to the board as evidence of 'ongoing security diligence'.

4. Permission Scoping — Fine-Grained Access

Some permissions support allow and deny scopes for path-based restrictions:

// capabilities/default.json — path-scoped permissions:
{
  "permissions": [
    // Allow reading files ONLY from specific paths:
    {
      "identifier": "fs:allow-read-file",
      "allow": [
        { "path": "$HOME/Documents/**" },  // home directory docs
        { "path": "$APPDATA/**" }           // app data directory
      ]
    },
    // Allow writing ONLY to the app data dir:
    {
      "identifier": "fs:allow-write-file",
      "allow": [
        { "path": "$APPDATA/**" }
      ]
    },
    // Explicitly deny access to sensitive paths:
    {
      "identifier": "fs:allow-read-file",
      "deny": [
        { "path": "$HOME/.ssh/**" },        // never touch SSH keys
        { "path": "$HOME/.aws/**" }         // never touch AWS credentials
      ]
    }
  ]
}

Path variables like $HOME, $APPDATA, $TEMP are resolved by Tauri at runtime for the current platform.

✅ True or False
Tauri v2's capabilities system makes it impossible for a successful XSS attack in the webview to access files that aren't declared in the permissions.
True! This is the core value proposition of Tauri's security model. The capabilities system is enforced at the IPC boundary in Rust — injected JavaScript can only call what's been declared. If APInox never declares fs:allow-read-file, no JS code (malicious or otherwise) can read arbitrary files. Compare to Electron where a successful XSS attack often gets full Node.js access to the whole OS.
Salesforce's permission model is: Profiles (baseline access), Permission Sets (additive), Permission Set Groups (bundled sets), Muting Permission Sets (take away from groups), Field-Level Security, Object-Level Security, Record-Level Security (Sharing Rules, Sharing Sets, Manual Sharing), and Restriction Rules. It is possible to give a user access to an object but not to any of its records. This is called 'enterprise security'. It took 3 years of Trailhead to understand. The consultants bill hourly.

5. Custom Command Permissions

Your own #[tauri::command] functions can also be wrapped in the permissions system. Tauri generates permission descriptors automatically during build:

// Annotate commands with their permission identifier:
#[tauri::command]
// Permission: "my-plugin:allow-save-project" (auto-generated)
pub async fn save_project(
    project: ApinoxProject,
    dir_path: String,
) -> Result<(), String> {
    FolderProjectStorage::save(&PathBuf::from(dir_path), &project)
        .map_err(|e| e.to_string())
}
// Then in capabilities:
{
  "permissions": [
    "my-plugin:allow-save-project",
    "my-plugin:allow-load-project"
    // "my-plugin:allow-delete-project" — not listed = not allowed
  ]
}
APInox currently doesn't use per-command permissions (all commands are implicitly allowed). For an internal desktop tool, this is fine. For a Tauri app distributed to untrusted users (browser-like threat model), you'd want per-command granularity.
🔗 Match It
Match the security mechanism to what it protects against:
All matched! These are layered defenses: CSP operates at the HTML/JS level (browser's built-in security), while capabilities operate at the IPC level (Rust-enforced security). Even if CSP is bypassed, capabilities are still enforced. Even if capabilities are overly broad, path scoping limits the blast radius.

6. withGlobalTauri — Convenient but Risky

// tauri.conf.json — APInox uses this:
{
  "app": {
    "withGlobalTauri": true
  }
}

When withGlobalTauri: true, Tauri injects window.__TAURI__ into every page load. This means your webview JavaScript can call window.__TAURI__.core.invoke() directly without importing from @tauri-apps/api.

The tradeoff:

  • ✅ Convenient for quick calls
  • ✅ No import overhead in the bundle
  • ❌ window.__TAURI__ is accessible to ANY injected script (XSS can call it directly)
  • ❌ Less type-safe than explicit imports
// How APInox handles the withGlobalTauri scenario:
// App.tsx — check if Tauri is available:
try {
    const { invoke } = await import('@tauri-apps/api/core');
    const os = await invoke<string>('get_platform_os');
    setPlatform(os);
} catch {
    console.log('⚠️ Tauri not detected (running in browser or invoke failed)');
}
🔮 Predict & Reveal
If you set "withGlobalTauri": false in tauri.conf.json but your TypeScript code uses window.__TAURI__.core.invoke() directly — what happens?
Runtime error: Cannot read properties of undefined (reading 'core'). With withGlobalTauri: false, window.__TAURI__ is undefined. Accessing .core on it throws. The fix is to use the proper module import: import { invoke } from '@tauri-apps/api/core' — this works regardless of the withGlobalTauri setting because it imports from the NPM package at bundle time, not from the runtime global. APInox's bridge.ts does exactly this — it dynamically imports the module rather than relying on the global.
